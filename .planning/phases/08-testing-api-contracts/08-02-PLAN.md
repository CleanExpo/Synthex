  <files>tests/unit/api/auth-routes.test.ts</files>
  <action>
Create comprehensive tests for authentication API routes. This extends beyond the existing basic auth-login.test.ts with full route contract testing.

**Setup:**
- Mock `@/lib/prisma` with mockPrisma (user.findUnique, user.create, session.create, etc.)
- Mock `@/lib/supabase-client` with mockSupabase (auth.signInWithPassword, auth.signUp)
- Mock `@/lib/auth/jwt-utils` (generateToken, verifyToken, getUserIdFromRequestOrCookies)
- Mock `@/lib/middleware/api-rate-limit` (authStrict, authGeneral → pass through to handler)
- Create mock NextRequest helper

**Test categories:**

1. **POST /api/auth/login — Full contract:**
   - Valid credentials → 200 with token and user object shape
   - Invalid email format → 400 validation error
   - Missing password → 400 validation error
   - User not found → 401 (not 404, don't leak user existence)
   - Wrong password (Supabase auth fails) → 401
   - OAuth-only user trying password login → 401 with hint message
   - Response shape: { success, token, user: { id, email, name, emailVerified } }

2. **POST /api/auth/signup — Full contract:**
   - Valid signup data → 201 with user object
   - Duplicate email → 409 conflict
   - Invalid email → 400
   - Weak password → 400
   - Creates user in both Supabase and Prisma
   - Response shape: { success, user, message }

3. **POST /api/auth/refresh (if exists):**
   - Valid refresh token → 200 with new token
   - Expired token → 401
   - Invalid token → 401

4. **POST /api/auth/logout:**
   - Authenticated → clears session, returns 200
   - Unauthenticated → still returns 200 (idempotent)

5. **Cross-auth patterns:**
   - All auth routes accept JSON body
   - Error responses never leak internal details
   - Rate limiting integrated (mock passes through)

**Test count target:** 18-22 test cases.

Don't: Duplicate the existing auth-login.test.ts basic validation tests.
Don't: Test Supabase internals — mock the auth responses.
Don't: Test rate limiting logic — it's tested in Phase 4.
Reference: Read app/api/auth/login/route.ts and signup/route.ts for exact validation and flow.

**Test categories:**

1. **GET /api/user/profile:**
   - Authenticated → returns user profile shape (id, name, email, avatar, createdAt)
   - Unauthenticated → 401
   - User not found in DB → 404

2. **PUT /api/user/profile:**
   - Valid update (name, bio) → 200 with updated profile
   - Invalid fields → 400
   - Unauthenticated → 401
   - Can't change email via profile update

3. **GET /api/user/subscription:**
   - Returns subscription details (plan, status, expiresAt, usage)
   - Free tier user → returns free plan defaults
   - Unauthenticated → 401

4. **GET /api/user/usage:**
   - Returns usage limits and current counts
   - Shape: { limits: { socialAccounts, aiPosts, personas }, current: { ... } }
   - Unauthenticated → 401

5. **PUT /api/user/settings:**
   - Valid settings update → 200
   - Invalid settings → 400
   - Unauthenticated → 401

6. **DELETE /api/user/account:**
   - Authenticated → deletes user, returns 200
   - Requires confirmation (body.confirm === true)
   - Unauthenticated → 401

**Test count target:** 15-18 test cases.

Don't: Test subscription billing logic — that's Plan 03.
Don't: Test file upload mechanics — just the route contract.
