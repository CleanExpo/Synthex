<task type="auto">
  <name>Task 1: Create social posting API route contract tests</name>
  <files>tests/unit/api/social-post.test.ts</files>
  <action>
Create tests for the unified social posting API route and platform-specific posting.

**Setup:**
- Mock `@/lib/prisma` (platformConnection.findMany, post.create, campaign.create, campaign.findFirst)
- Mock `@/lib/auth/jwt-utils` (getUserIdFromRequestOrCookies)
- Mock `@/lib/social/index` (createPlatformService → mock service with createPost)
- Mock field encryption module (decrypt → returns test tokens)
- Mock rate limiting (pass through)

**Test categories:**

1. **POST /api/social/post — Input validation:**
   - Valid post with single platform → 200 with results
   - Valid post with multiple platforms → 200 with per-platform results
   - Missing content → 400
   - Empty platforms array → 400
   - Invalid platform name → 400
   - Unauthenticated → 401

2. **POST /api/social/post — Multi-platform posting:**
   - All platforms succeed → { success: true, results: [...] }
   - One platform fails → partial success with error for failed platform
   - All platforms fail → { success: false, errors: [...] }
   - Each result has: { platform, success, postId, url, message }

3. **POST /api/social/post — Platform connection handling:**
   - No connection for requested platform → error for that platform
   - Expired token → attempts refresh, then posts
   - Connection with encrypted tokens → decrypts before posting

4. **POST /api/social/post — Campaign integration:**
   - No campaignId → auto-creates campaign
   - Existing campaignId → links post to campaign
   - Campaign creation failure → still posts (non-blocking)

5. **POST /api/social/post — Scheduling:**
   - scheduledAt in future → creates scheduled post, doesn't post immediately
   - scheduledAt in past → rejects with 400

6. **Platform-specific route (e.g., twitter/post):**
   - Valid Twitter post → 200
   - Platform-specific validation (Twitter character limit)
   - Response matches unified contract shape

**Test count target:** 18-22 test cases.

Don't: Test platform service internals — already tested in Phase 7.
Don't: Test encryption internals — mock decrypt to return test values.
Reference: Read app/api/social/post/route.ts for exact validation, multi-platform flow, and response shapes.

<task type="auto">
  <name>Task 2: Create critical path integration tests</name>
  <files>tests/integration/critical-paths.test.ts</files>
  <action>
Create integration-style tests that validate the two most critical cross-service flows. These test the contract between services (not HTTP transport) — similar to the existing tests/integration/ pattern.

**Setup:**
- Mock Prisma with stateful in-memory tracking (create → store, findFirst → retrieve)
- Mock Stripe SDK
- Mock platform services
- Mock auth/JWT

**Test categories:**

1. **Payment lifecycle flow:**
   - Create checkout session → receive subscription.created webhook → subscription active in DB
   - Active subscription → receive subscription.updated (plan change) → subscription updated in DB
   - Active subscription → receive subscription.cancelled → subscription marked cancelled
   - Invoice payment succeeded → subscription confirmed active
   - Invoice payment failed → subscription marked past_due
   - Verify subscription state transitions are consistent

2. **Usage limit enforcement flow:**
   - Free user → check limit for socialAccounts → allowed (under 2)
   - Free user → check limit for 3rd socialAccount → denied
   - User upgrades to professional → check limit for 5 accounts → allowed
   - User posts → usage incremented → check at limit → denied
   - Monthly reset → usage cleared → check again → allowed

3. **Social posting with subscription check:**
   - User with active subscription → post succeeds
   - User posts → usage incremented correctly
   - User at limit → post denied with upgrade prompt

4. **Webhook idempotency:**
   - Same webhook event processed twice → no duplicate state changes
   - Events arriving out of order → final state is correct

**Test count target:** 15-18 test cases.

Don't: Start a dev server — these are programmatic integration tests.
Don't: Use supertest/HTTP — call service methods directly.
Don't: Create Playwright E2E tests — defer browser-based E2E to when CI infrastructure supports it.
Reference: Read lib/stripe/subscription-service.ts for subscription state machine and usage limit logic.
