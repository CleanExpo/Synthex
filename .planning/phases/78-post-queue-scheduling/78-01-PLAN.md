# Plan 78-01: Queue Management Dashboard & Bulk Actions

## objective

Build a dedicated post queue management page at `/dashboard/schedule/queue` with a filterable, sortable post queue list, bulk selection/actions, and drag-to-reorder capability. Currently, the schedule page (`/dashboard/schedule`) shows a calendar view of scheduled posts, but there is no queue-oriented view where users can see all pending posts in a flat list, select multiple posts for bulk rescheduling/deletion/publishing, or reorder their publishing priority. The calendar page (`/dashboard/calendar`) has a basic schedule modal but lacks queue management. This plan adds the queue-centric view that powers the bulk scheduling workflow in 78-02.

Linear: SYN-44

## context

### Discovery Findings

**Gap 1 -- No queue view**: The schedule page has week, month, and list views. The list view (`components/schedule/list-view.tsx`) shows posts sorted by scheduledFor but has no checkbox selection, no bulk actions, and no drag reorder. Users must click into each post individually to manage it.

**Gap 2 -- No bulk operations**: There is no way to select multiple posts and reschedule, delete, or change their status in one action. The scheduler API supports individual PATCH/DELETE operations but no batch endpoint.

**Gap 3 -- No queue priority/ordering**: Posts are processed by the publish cron in `scheduledAt ASC` order, but the user has no visibility or control over this ordering. If a user has 10 posts scheduled for the same hour, there is no way to prioritise which publishes first.

**Gap 4 -- No failed post retry from queue**: When a post fails to publish, it is marked `status: 'failed'` with `metadata.publishError`. The schedule page shows the failed status but has no retry mechanism beyond manually editing and re-scheduling the post.

### Architecture Decision

- **Queue page**: New page at `app/dashboard/schedule/queue/page.tsx` -- flat list view with checkbox selection, bulk action toolbar, and status/platform filters
- **Bulk API**: New `POST /api/scheduler/posts/bulk` endpoint for batch operations (reschedule, delete, set-status)
- **Queue components**: `components/queue/` -- `QueueTable`, `QueueBulkActions`, `QueueFilters`, `QueueStats`
- **Navigation**: Add "Queue" tab/link to the schedule header or sidebar
- **SWR**: Use SWR for data fetching with `mutate` for optimistic updates after bulk operations

## tasks

### Task 1: Bulk scheduler API -- `app/api/scheduler/posts/bulk/route.ts`

Create a batch operations endpoint for queue management.

**POST /api/scheduler/posts/bulk**

Request body (Zod schema):
```ts
const bulkActionSchema = z.object({
  action: z.enum(['reschedule', 'delete', 'set-status', 'retry']),
  postIds: z.array(z.string()).min(1).max(50),
  // For reschedule:
  scheduledAt: z.string().datetime().optional(),
  // For reschedule with offset (shift all by N hours):
  offsetHours: z.number().optional(),
  // For set-status:
  status: z.enum(['draft', 'scheduled', 'cancelled']).optional(),
});
```

**Actions:**
1. `reschedule` -- Update `scheduledAt` for all selected posts. If `scheduledAt` provided, set all to that time. If `offsetHours` provided, shift each post's existing `scheduledAt` by that many hours.
2. `delete` -- Delete all selected posts (verify ownership per post).
3. `set-status` -- Update status for all selected posts (e.g., pause by setting to 'draft', resume by setting to 'scheduled').
4. `retry` -- For failed posts: reset status to 'scheduled', clear `metadata.publishError` and `metadata.failedAt`, set `scheduledAt` to now + 5 minutes.

**Auth**: Use `getUserIdFromRequestOrCookies` from `@/lib/auth/jwt-utils`. Verify ownership of each post through its campaign's `userId`.

**Response**:
```json
{
  "success": true,
  "processed": 5,
  "failed": 0,
  "results": [
    { "id": "cuid_xxx", "status": "updated" },
    { "id": "cuid_yyy", "status": "updated" }
  ]
}
```

Use Prisma `$transaction` for atomicity on delete and set-status actions. For reschedule with offset, process sequentially to handle per-post date arithmetic.
