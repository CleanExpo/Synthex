# Plan 78-04: Queue Notifications, Retry Logic & Schedule Health

## objective

Close the feedback loop on scheduled post publishing by adding: (1) toast and in-app notifications when posts publish or fail, (2) automatic retry with exponential backoff for transient failures, (3) a schedule health dashboard widget showing queue vitals and publishing success rates, and (4) a post history log showing the full lifecycle of each post (created, scheduled, retried, published/failed). Currently, the publish cron marks posts as published or failed, but users only discover the outcome by manually checking the schedule page. There is no notification, no auto-retry, and no historical audit trail for post lifecycle events.

## context

### Discovery Findings

**Gap 1 -- No publish notifications**: When the cron publishes or fails a post, there is no notification sent to the user. The user must manually navigate to the schedule page to check status. For a platform that runs on autopilot, this is a critical gap.

**Gap 2 -- No auto-retry for transient failures**: The publish cron sets `status: 'failed'` permanently on the first failure. Many failures are transient (rate limits, temporary API outages, token refresh races). These should be automatically retried before being marked as permanently failed.

**Gap 3 -- No publishing success metrics**: There is no dashboard widget showing publishing health: success rate, average publish delay (time between scheduledAt and publishedAt), failure reasons breakdown. Users have no way to assess the reliability of their scheduling pipeline.

**Gap 4 -- No post lifecycle history**: The Post model only has `status` (current) and `metadata.publishError`/`metadata.failedAt` (last failure). There is no history of state transitions (when was it created? when was it rescheduled? how many retry attempts?).

### Architecture Decision

- **Retry logic in publish cron**: Add `retryCount` and `maxRetries` to post metadata. On transient failure, increment retry count and reschedule for retry (5 min, 15 min, 60 min backoff). Only mark as permanently failed after maxRetries (default 3).
- **In-app notifications**: Use the existing notification system (find it) or create `Notification` model entries on publish/fail events. Show via a notification bell widget.
- **Schedule health widget**: `components/queue/schedule-health.tsx` -- card showing 7-day success rate, average delay, failure breakdown.
- **Post lifecycle log**: Store state transitions in `metadata.history` array on the Post model. No schema change needed (JSON field).

## tasks

### Task 1: Add retry logic to publish cron

**File**: `app/api/cron/publish-scheduled/route.ts`

Update the failure handling path to support automatic retries.

**Changes to the post processing loop** (inside the `for (const post of duePosts)` block):

1. After a publish failure, check if the error is retryable:
   ```ts
   function isRetryableError(error: string): boolean {
     const retryablePatterns = [
       'rate limit', 'too many requests', '429',
       'timeout', 'ETIMEDOUT', 'ECONNRESET',
       'temporarily unavailable', '503', '502',
       'token refresh', 'refresh token',
     ];
     const lowerError = error.toLowerCase();
     return retryablePatterns.some(p => lowerError.includes(p));
   }
   ```

2. Read retry state from post metadata:
   ```ts
   const metadata = (post.metadata as Record<string, unknown>) || {};
   const retryCount = (metadata.retryCount as number) || 0;
   const maxRetries = 3;
   ```

3. If retryable AND retryCount < maxRetries, reschedule instead of failing:
   ```ts
   if (isRetryableError(errorMessage) && retryCount < maxRetries) {
     const backoffMinutes = [5, 15, 60][retryCount] || 60;
     const retryAt = new Date(Date.now() + backoffMinutes * 60 * 1000);

     await prisma.post.update({
       where: { id: post.id },
       data: {
         scheduledAt: retryAt,
         metadata: {
           ...metadata,
           retryCount: retryCount + 1,
           lastRetryError: errorMessage,
           lastRetryAt: new Date().toISOString(),
           history: [
             ...((metadata.history as unknown[]) || []),
             { event: 'retry_scheduled', at: new Date().toISOString(), reason: errorMessage, attempt: retryCount + 1, retryAt: retryAt.toISOString() }
           ],
         },
       },
     });

     results.push({ id: post.id, platform, status: 'failed', error: `Retry ${retryCount + 1}/${maxRetries} scheduled for ${retryAt.toISOString()}` });
     continue; // Don't count as a final failure
   }
   ```

4. If NOT retryable or maxRetries exceeded, mark as permanently failed (existing behaviour) but also add to history:
   ```ts
   await markPostFailed(post.id, errorMessage, {
     retryCount,
     finalFailure: true,
     history: [
       ...((metadata.history as unknown[]) || []),
       { event: 'failed_permanently', at: new Date().toISOString(), reason: errorMessage, attempts: retryCount }
     ],
   });
   ```

5. On successful publish, add a history entry:
   ```ts
   history: [
     ...((metadata.history as unknown[]) || []),
     { event: 'published', at: new Date().toISOString(), platformPostId: postResult.postId }
   ]
   ```

Update `markPostFailed` to accept optional extra metadata to merge.

### Task 3: Schedule health widget -- `components/queue/schedule-health.tsx`

Dashboard widget showing publishing pipeline health metrics.

**Data source**: New `GET /api/scheduler/stats` endpoint (or add a `stats` action to existing scheduler API).

**Stats to calculate** (server-side):
```ts
interface ScheduleHealthStats {
  last7Days: {
    total: number;
    published: number;
    failed: number;
    retrying: number;
    successRate: number; // percentage
  };
  averageDelayMinutes: number; // avg(publishedAt - scheduledAt) for published posts
  failureReasons: Array<{ reason: string; count: number }>; // top 5 failure reasons
  nextScheduled: {
    count: number;
    nextAt: string | null;
  };
  retryQueue: {
    count: number; // posts with retryCount > 0 and status = 'scheduled'
  };
}
```

**API**: `GET /api/scheduler/stats` -- Queries Post model with Prisma aggregations:
- Count by status where `createdAt >= 7 days ago`
- Average delay: `AVG(publishedAt - scheduledAt)` for published posts (use raw query or compute in JS)
- Failure reasons: group by `metadata.publishError` from failed posts
- Next scheduled: `Post.findFirst({ where: { status: 'scheduled' }, orderBy: { scheduledAt: 'asc' } })`
- Retry queue: count where `metadata.retryCount > 0` and status = 'scheduled'

**Widget rendering**:
1. Card titled "Schedule Health"
2. Primary metric: success rate as a large percentage with green/amber/red colour (>90% green, 70-90% amber, <70% red)
3. Secondary metrics in a grid:
   - Published: N | Failed: N | Retrying: N
   - Avg delay: Xm
   - Next up: "in 2h 15m" or "No posts scheduled"
4. Failure reasons: if any failures, show top 3 as a mini list with count badges

### Task 5: Scheduler stats API endpoint -- `app/api/scheduler/stats/route.ts`

Create the stats endpoint consumed by the schedule health widget.

**Auth**: Use `getUserIdFromRequestOrCookies`. Scope all queries to the user's campaigns.

**Implementation**:
```ts
const sevenDaysAgo = new Date(Date.now() - 7 * 24 * 60 * 60 * 1000);

// Get user's campaign IDs
const campaignIds = await getUserCampaignIds(userId);
const baseCampaignFilter = { campaignId: { in: campaignIds } };

// Count by status (last 7 days)
const [total, published, failed] = await Promise.all([
  prisma.post.count({ where: { ...baseCampaignFilter, createdAt: { gte: sevenDaysAgo } } }),
  prisma.post.count({ where: { ...baseCampaignFilter, status: 'published', publishedAt: { gte: sevenDaysAgo } } }),
  prisma.post.count({ where: { ...baseCampaignFilter, status: 'failed', updatedAt: { gte: sevenDaysAgo } } }),
]);

// Retry queue count
const retrying = await prisma.post.count({
  where: {
    ...baseCampaignFilter,
    status: 'scheduled',
    metadata: { path: ['retryCount'], gt: 0 },
  },
});

// Average publish delay
const publishedPosts = await prisma.post.findMany({
  where: {
    ...baseCampaignFilter,
    status: 'published',
    publishedAt: { gte: sevenDaysAgo },
    scheduledAt: { not: null },
  },
  select: { scheduledAt: true, publishedAt: true },
  take: 100,
});

const avgDelayMs = publishedPosts.reduce((sum, p) => {
  if (p.publishedAt && p.scheduledAt) {
    return sum + (p.publishedAt.getTime() - p.scheduledAt.getTime());
  }
  return sum;
}, 0) / Math.max(publishedPosts.length, 1);

// Next scheduled
const nextScheduled = await prisma.post.findFirst({
  where: { ...baseCampaignFilter, status: 'scheduled', scheduledAt: { gte: new Date() } },
  orderBy: { scheduledAt: 'asc' },
  select: { scheduledAt: true },
});

// Top failure reasons (parse from metadata)
const failedPosts = await prisma.post.findMany({
  where: {
    ...baseCampaignFilter,
    status: 'failed',
    updatedAt: { gte: sevenDaysAgo },
  },
  select: { metadata: true },
  take: 50,
});
```

Group failure reasons by `metadata.publishError` and return top 5.

**Response**: Return `ScheduleHealthStats` shape.
