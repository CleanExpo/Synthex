## Architecture Notes

### Data Fetching Pattern

Per `CLAUDE.md`, client components use `useSWR` with `credentials: 'include'`.
Reference implementation: `components/dashboard/GamificationWidget.tsx`.

```typescript
// Pattern for all data fetching in this phase
import useSWR from 'swr';

const fetchJson = (url: string) =>
  fetch(url, { credentials: 'include' }).then((r) => r.json());

const { data, isLoading, error } = useSWR('/api/bayesian/spaces', fetchJson);
```

### API Endpoints Used by UI

All routes were built in Phase 102:

| Endpoint | Method | Purpose |
|----------|--------|---------|
| `/api/bayesian/spaces` | GET | List all BO spaces for the org |
| `/api/bayesian/run` | POST | Trigger an optimisation run |
| `/api/bayesian/status/[jobId]` | GET | Poll run status |

The observe and suggest endpoints are server-to-server only — not called from
the dashboard UI directly.

### Subscription Gating

Free-tier users (`optimisationSpaces: 0`) see the upgrade prompt.
Pro+ users see the dashboard.

Use `useSubscription()` from `hooks/useSubscription.ts` (same hook used by
`GEOFeatureGate`) to check plan access. The required plan is `'pro'` (not
`'professional'` — BO is available from Pro upward per `feature-limits.ts`).

Note: `GEOFeatureGate` uses `requiredPlan: 'professional'` — BO uses `'pro'`.
`useSubscription().hasAccess()` accepts the plan string, so pass `'pro'`.

### Design System

Match the glassmorphic pattern from `app/dashboard/geo/page.tsx`:
- Cards: `bg-surface-base/80 backdrop-blur-xl border border-cyan-500/10`
- Headers: `text-white`, section labels `text-gray-400`
- Accent colour for BO: violet/purple (`text-violet-400`, `border-violet-500/10`)
  to differentiate from GEO's cyan
- Icons: import from `@/components/icons` (barrel, never lucide-react directly)
- Status badges: use Radix `Badge` component from `@/components/ui/badge`

### Task 5 — Optimisation dashboard page

**File**: `app/dashboard/optimisation/page.tsx`

Main client page that assembles the dashboard.

```typescript
'use client';

import useSWR from 'swr';
import { BOFeatureGate } from '@/components/bayesian/BOFeatureGate';
import { OptimisationSpaceCard } from '@/components/bayesian/OptimisationSpaceCard';
import { RunHistoryTable } from '@/components/bayesian/RunHistoryTable';
import { Brain, Plus } from '@/components/icons';
```

**Data fetching** (useSWR, `credentials: 'include'`):

```typescript
const fetchJson = (url: string) =>
  fetch(url, { credentials: 'include' }).then((r) => r.json());

const { data: spacesData, isLoading: spacesLoading, mutate: mutateSpaces } =
  useSWR('/api/bayesian/spaces', fetchJson, { refreshInterval: 30_000 });

const { data: runsData, isLoading: runsLoading, mutate: mutateRuns } =
  useSWR('/api/bayesian/run', fetchJson, { refreshInterval: 10_000 });
```

Note: the runs endpoint at `/api/bayesian/run` only has a POST method (trigger).
To list runs, a separate GET on `/api/bayesian/run` will need to be checked.
If the GET is not available, the RunHistoryTable receives an empty array with
a comment explaining the endpoint gap — do NOT invent or fabricate run list data.
Check `app/api/bayesian/run/route.ts` before implementing — if GET is missing,
the table shows an empty state and a note: "Run history available after first
optimisation run."

**Run optimisation handler:**

```typescript
const [runningSpaceId, setRunningSpaceId] = useState<string | null>(null);

const handleRunOptimisation = async (spaceId: string) => {
  setRunningSpaceId(spaceId);
  try {
    const response = await fetch('/api/bayesian/run', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      credentials: 'include',
      body: JSON.stringify({ spaceId }),
    });
    if (!response.ok) {
      const err = await response.json();
      // Surface error to user (toast or inline error)
    } else {
      void mutateSpaces();
      void mutateRuns();
    }
  } finally {
    setRunningSpaceId(null);
  }
};
```

**Page layout:**

```
<BOFeatureGate>
  <div className="space-y-6">
    <!-- Header -->
    <div className="flex items-center justify-between">
      <h1>AI Optimisation</h1>
      <p>Bayesian Optimisation — adaptive parameter tuning per organisation</p>
    </div>

    <!-- Spaces grid -->
    <section>
      <h2>Active Optimisation Spaces</h2>
      {spacesLoading ? <skeleton> : <grid of OptimisationSpaceCards>}
      {spaces.length === 0 ? <empty state with info about creating spaces> : null}
    </section>

    <!-- Run history -->
    <section>
      <h2>Run History</h2>
      <RunHistoryTable runs={runs} isLoading={runsLoading} />
    </section>

</BOFeatureGate>
```

Spaces grid: `grid grid-cols-1 md:grid-cols-2 lg:grid-cols-3 gap-4`

Empty spaces state: show an info card explaining that optimisation spaces are
created automatically as the user's organisation uses GEO analysis. No manual
creation button — spaces are auto-provisioned by the API.

Loading skeleton: `animate-pulse` divs matching card heights, 2 per row.

**API response handling:**
- `spacesData?.data` is `BOSpace[]` from GET `/api/bayesian/spaces`
- Guard against undefined: `const spaces = spacesData?.data ?? []`

## Risk Notes

**Risk 1 — GET runs endpoint may not exist**
`app/api/bayesian/run/route.ts` only has a POST handler. If GET is absent,
`RunHistoryTable` will receive no data. The task description handles this with
an empty-state message. Do NOT create a fake GET endpoint or fabricate data.
If needed, file this as a follow-up task for Phase 103.

**Risk 2 — weightSource field absent from GEO analyze response**
The `source` field from `getGeoScoreWeights()` may not be forwarded to the
API response. Read `app/api/geo/analyze/route.ts` before Task 8. If absent,
skip the badge display and note the gap.

**Risk 3 — Brain icon already used in sidebar**
`Brain` is used for Psychology nav item. Using it again for Optimisation is
fine — duplicate use of an icon in different nav items is acceptable.

**Risk 4 — hasAccess plan name mismatch**
`useSubscription().hasAccess('pro')` — verify that the plan name `'pro'` matches
the DB values in `BO_PLAN_LIMITS` (`feature-limits.ts` uses `'pro'`).
`GEOFeatureGate` uses `'professional'` which may be a different plan tier.
Do not assume — read `hooks/useSubscription.ts` before implementing the gate.
