## Prisma Schema Changes

Add to `prisma/schema.prisma` (after existing models):

```prisma
model WorkflowExecution {
  id               String   @id @default(cuid())
  organizationId   String
  workflowId       String?
  status           String   @default("pending")
  // pending | running | waiting_approval | completed | failed | cancelled
  title            String
  triggerType      String   @default("manual")
  // manual | scheduled | api
  triggeredBy      String
  currentStepIndex Int      @default(0)
  totalSteps       Int      @default(0)
  inputData        Json?
  outputData       Json?
  errorMessage     String?
  startedAt        DateTime?
  completedAt      DateTime?
  createdAt        DateTime @default(now())
  updatedAt        DateTime @updatedAt
  organization     Organization    @relation(fields: [organizationId], references: [id], onDelete: Cascade)
  stepExecutions   StepExecution[]

  @@index([organizationId])
  @@index([organizationId, status])
  @@map("workflow_executions")
}

model StepExecution {
  id                  String   @id @default(cuid())
  workflowExecutionId String
  stepIndex           Int
  stepName            String
  stepType            String
  // ai | approval | action | validation
  status              String   @default("pending")
  // pending | running | completed | failed | skipped | waiting_approval
  inputData           Json?
  outputData          Json?
  confidenceScore     Float?
  autoApproved        Boolean  @default(false)
  approvedBy          String?
  approvedAt          DateTime?
  errorMessage        String?
  retryCount          Int      @default(0)
  startedAt           DateTime?
  completedAt         DateTime?
  createdAt           DateTime @default(now())
  workflowExecution   WorkflowExecution @relation(fields: [workflowExecutionId], references: [id], onDelete: Cascade)

  @@index([workflowExecutionId])
  @@index([workflowExecutionId, stepIndex])
  @@map("step_executions")
}
```

**Backward compatibility**: Both new models. No changes to existing models.

## Files to Create

### `lib/workflow/index.ts`
Barrel export for all workflow library exports.

### `lib/workflow/types.ts`
TypeScript types for the workflow system:
- `WorkflowStepDefinition` — blueprint for a single step
- `WorkflowDefinition` — full workflow template
- `StepContext` — assembled context passed to an AI step
- `StepResult` — what a step executor returns `{ output, confidenceScore?, error? }`

### `lib/workflow/orchestrator.ts`
The central coordinator. Deterministic logic — no AI calls here.

Key responsibilities:
- `advanceWorkflow(executionId)`: load execution, find next step, call step-executor
- `handleStepResult(stepExecutionId, result)`: store result, check gate, enqueue next step
- `checkGate(stepExecution, template)`: auto-approve if confidence ≥ threshold, else wait
- `cancelExecution(executionId)`: mark cancelled at current step boundary
- 2-retry cap: if `retryCount >= 2`, mark step as failed, surface to human (do not loop)
- Updates `WorkflowExecution.currentStepIndex` and `status` atomically

### `lib/workflow/step-executor.ts`
Executes a single step by its `stepType`.

Key responsibilities:
- `executeStep(stepDef, context)`: routes to correct step-type handler
- Wraps execution in try/catch, returns structured `StepResult`
- Handles timeout: steps have a max execution time (configurable, default 30s)
- Does NOT contain AI logic — delegates to step-types/

### `lib/workflow/context-builder.ts`
Assembles context for AI steps. Implements the Minions "context assembly pipeline."

Key responsibilities:
- `buildStepContext(execution, stepIndex)`: gather relevant prior outputs
- Token budget: include last N step outputs (default: last 3), strip verbose content
- Pattern: step definition + previous StepExecution.outputData (not full history)
- Returns `StepContext` typed object

## Key Design Decisions

**Why orchestrator is deterministic**: Following Minions principle 2 ("the system runs the
model"). Routing logic, gate logic, and retry logic live in TypeScript — not in LLM output.
The LLM responds to prompts; the orchestrator decides what happens with the response.

**Why context-builder has a token budget**: Following Minions principle 1 ("context > model")
and principle 5 ("walls before models"). We invest in giving each step exactly the right
context — not everything, just the relevant prior outputs. This prevents context drift and
keeps costs predictable.

**Why 2-retry cap**: Following Minions principle 5. If a step fails twice, the system has
a problem the LLM can't solve by trying again. Surface to human.
