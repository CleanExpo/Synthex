## Step Type Implementations

All step types live in `lib/workflow/step-types/`. Each exports:
```typescript
export async function execute(
  stepDef: WorkflowStepDefinition,
  context: StepContext
): Promise<StepResult>
```

### `ai-generate.ts`
Content generation. Calls OpenRouter via the content pipeline.
- Builds prompt from `stepDef.promptTemplate` + context
- Requests structured output including `confidenceScore` (0.0–1.0)
- Returns `{ output: { content, metadata }, confidenceScore }`

### `ai-analyse.ts`
Classification, scoring, summarisation.
- No content creation — analysis of existing content in context
- Returns `{ output: { analysis, score, reasoning }, confidenceScore }`

### `ai-enrich.ts`
Adds metadata, hashtags, CTAs to existing content.
- Input: content from previous step
- Returns `{ output: { enrichedContent, additions }, confidenceScore }`

### `human-approval.ts`
Pauses the workflow for human decision.
- Does NOT call AI
- Creates a notification (via existing notification system if available)
- Returns `{ output: { waitingFor: 'human_approval' }, requiresApproval: true }`
- Orchestrator marks step as `waiting_approval` and pauses queue

### `action-publish.ts`
Pushes content to a social platform.
- Calls the appropriate platform service from `lib/social/`
- Returns `{ output: { platformPostId, publishedAt }, confidenceScore: 1.0 }`
- Human approval gate is ALWAYS enforced before this step type executes

### `action-schedule.ts`
Adds content to the content calendar.
- Creates a scheduled post in Prisma
- Returns `{ output: { scheduledPostId, scheduledAt }, confidenceScore: 1.0 }`
- Human approval gate enforced before this step type

### `action-notify.ts`
Sends a team notification.
- Uses existing notification infrastructure
- Returns `{ output: { notificationSent: boolean }, confidenceScore: 1.0 }`

## BullMQ Integration

### Add to `lib/queue/bull-queue.ts`

```typescript
// New job type
export interface WorkflowStepJob {
  type: 'workflow:execute-step';
  workflowExecutionId: string;
  stepIndex: number;
  retryCount: number;
}
```

### New worker `lib/queue/workers/workflow-step.worker.ts`

```typescript
// Processes workflow:execute-step jobs
// Calls orchestrator.advanceWorkflow(job.data.workflowExecutionId)
// On success: orchestrator enqueues next step automatically
// On failure: retryCount < 2 → re-enqueue; retryCount >= 2 → mark failed
```

### Job flow

```
POST /api/workflows/executions
  → creates WorkflowExecution (status: pending)
  → enqueues { type: 'workflow:execute-step', workflowExecutionId, stepIndex: 0, retryCount: 0 }

Worker picks up job
  → orchestrator.advanceWorkflow(executionId)
  → marks execution running, step running
  → context-builder assembles step context
  → step-executor executes step
  → result stored in StepExecution
  → gate check: auto-approve or waiting_approval
  → if auto-approved: enqueue next step job
  → if waiting_approval: pause (no new job — approval route re-enqueues)
  → if all steps done: mark execution completed
```

## WorkflowTemplate Model (needed by templates route)

Add to Prisma schema:
```prisma
model WorkflowTemplate {
  id                   String   @id @default(cuid())
  organizationId       String
  name                 String
  description          String?
  steps                Json     // WorkflowStepDefinition[]
  autoApproveThreshold Float    @default(0.85)
  isActive             Boolean  @default(true)
  createdBy            String
  createdAt            DateTime @default(now())
  updatedAt            DateTime @updatedAt
  organization         Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)

  @@index([organizationId])
  @@map("workflow_templates")
}
```

Also add the `Organization` side relation:
```prisma
workflowTemplates    WorkflowTemplate[]
workflowExecutions   WorkflowExecution[]
```
