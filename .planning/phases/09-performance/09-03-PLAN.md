- `ApiUsage`: `@@index([userId, createdAt])` — usage tracking

**Priority 3 — Foreign key indexes (Prisma doesn't auto-index foreign keys in PostgreSQL):**
- `CompetitorSnapshot`: `@@index([competitorId])` — FK to TrackedCompetitor
- `CompetitorPost`: `@@index([competitorId])` — FK to TrackedCompetitor
- `CompetitorAlert`: `@@index([competitorId])` — FK to TrackedCompetitor
- `ABTestVariant`: `@@index([testId])` — FK to ABTest
- `ABTestResult`: `@@index([variantId])` — FK to ABTestVariant
- `ReportDelivery`: `@@index([reportId])` — FK to Report
- `ContentComment`: `@@index([shareId])` — FK to ContentShare
- `AIMessage`: `@@index([conversationId])` — FK to AIConversation

**Format:** Add `@@index` directives inside the model block, after all field definitions and before the closing brace. Group them after any existing `@@unique` or `@@map` directives.

Example:
```prisma
model Post {
  id        String   @id @default(uuid())
  userId    String
  // ... fields ...

  @@index([userId])
  @@index([userId, createdAt])
  @@index([userId, platform])
  @@map("posts")
}
```

**Important:** Only add indexes where the field actually exists on the model. Read each model definition first. Some models may use different field names (e.g., `ownerId` instead of `userId`, or `authorId`).

**Do NOT:**
- Add indexes on fields that are already `@unique` (Prisma auto-indexes unique fields)
- Add indexes on `@id` fields (already indexed)
- Change any field definitions or types
- Add/remove any models

**After adding indexes:**
- Run `npx prisma validate` to verify schema is valid
- Run `npx prisma generate` to regenerate client

**1. Dashboard stats (app/api/dashboard/stats/route.ts):**
- Read the route and identify all Prisma queries
- Check for N+1 patterns (e.g., fetching user, then separately fetching campaigns, posts, etc.)

- Check if competitors are fetched with includes or if there are N+1 queries for snapshots

- Look for: unnecessary data fetching, missing select clauses, queries inside loops that could be batched

**3. Notifications (app/api/notifications/route.ts):**
- Check if unread count + notification list are separate queries that could be combined
- Verify pagination is implemented (not fetching all notifications)
- Add `take` limit if missing (default to 50)
- Ensure `select` only returns needed fields (id, type, title, message, read, createdAt — NOT full user objects)

**For each route:**
- Read the file first

- Apply optimizations:
  - `select` for limiting returned fields
  - `take` for pagination limits
  - `Promise.all()` for independent parallel queries
  - `include` with nested `select` for related data
- Only modify queries where there's a clear improvement
- Do NOT refactor unrelated code
