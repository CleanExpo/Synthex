# Plan 75-01: God Mode Admin Panel

## objective

Wire the existing admin panel to real APIs, add owner-only route protection, and add platform health + audit log views. The panel exists but has three critical issues: (1) it calls `supabase.auth.admin.listUsers()` client-side (security bug), (2) user actions are stub toasts with no actual backend calls, and (3) anyone can navigate to `/dashboard/admin/` without an owner check.

Linear: SYN-18
Phase dir: `.planning/phases/75-god-mode-admin/`

---

## execution_context

- Auth pattern: `isOwnerEmail()` in `lib/auth/jwt-utils.ts` ‚Äî checks owner email set
- Existing API: `app/api/admin/users/route.ts` ‚Äî GET (list/search/paginate) + POST (suspend/activate/delete), ADMIN_API_KEY auth
- Existing API: `app/api/admin/audit-log/route.ts` ‚Äî GET with filters
- Existing API: `app/api/admin/jobs/route.ts` ‚Äî GET queue stats + POST actions (retry/cancel)
- Existing UI: `components/admin/` ‚Äî types, header, stats, search, bulk-actions, table, edit-dialog
- Existing page: `app/dashboard/admin/page.tsx` ‚Äî 'use client', broken (see objective)
- Dashboard layout: `app/dashboard/layout.tsx` ‚Äî contains sidebar nav

## context

### Discovery Findings

**`app/dashboard/admin/page.tsx` issues:**
- `supabase.auth.admin.listUsers()` is a service-role-only Supabase call ‚Äî never safe in browser
- `handleUserAction('ban'/'unban'/'reset-password')` ‚Üí only shows toast, no API call
- `handleSaveUser` ‚Üí `await new Promise(resolve => setTimeout(resolve, 800))` ‚Äî pure fake delay
- Does NOT call `/api/admin/users` ‚Äî bypasses the real Prisma API entirely

**`app/api/admin/users/route.ts`** (already correct):
- GET: list users from Prisma with search/pagination, returns `{ success, data, pagination }`
- POST: suspend/activate/delete via Prisma transaction + AuditLog entry
- Auth: `x-admin-api-key` header OR JWT token with `preferences.role === 'admin'`

**`lib/auth/jwt-utils.ts`**:
- `isOwnerEmail(email)` ‚Äî returns true for owner email addresses
- Owner emails: `phill.mcgurk@gmail.com`, `phill.mcgurk+test1@gmail.com`

**No middleware protection** on `/dashboard/admin/` ‚Äî any logged-in user can access it.

### Architecture Decision

- Route guard: Add `app/dashboard/admin/layout.tsx` ‚Äî server component, reads JWT, checks `isOwnerEmail()`, redirects to `/dashboard` if not owner
- Admin page rewrite: Replace client-side Supabase admin calls with SWR ‚Üí `/api/admin/users`
- Actions: Call POST `/api/admin/users` with `x-admin-api-key` header (from `NEXT_PUBLIC_ADMIN_API_KEY` ‚Äî env var to add) OR rely on JWT role check
- Platform health tab: New tab showing subscription stats (from `/api/stats/platform` or direct Prisma query via a new endpoint), job queue stats from `/api/admin/jobs`
- Audit log tab: New tab showing paginated audit logs from `/api/admin/audit-log`

**Note on ADMIN_API_KEY from client**: The API currently uses `x-admin-api-key` header. For the dashboard (browser context), we should use the JWT auth path (user is logged in as owner ‚Üí JWT has userId ‚Üí API checks `preferences.role`). We need to ensure the owner user has `preferences.role: 'admin'` or add owner email check to the admin API verifyAdmin function.

## tasks

### Task 1: Owner route guard ‚Äî `app/dashboard/admin/layout.tsx`

Create a server component layout that protects the entire `/dashboard/admin/` subtree.

**File**: `app/dashboard/admin/layout.tsx`

```tsx
import { redirect } from 'next/navigation';
import { getServerSession } from '@/lib/auth/auth-service';
import { isOwnerEmail } from '@/lib/auth/jwt-utils';

export default async function AdminLayout({ children }: { children: React.ReactNode }) {
  const session = await getServerSession();
  if (!session?.user?.email || !isOwnerEmail(session.user.email)) {
    redirect('/dashboard');
  }
  return <>{children}</>;
}
```

First check how `getServerSession` works in `lib/auth/auth-service.ts` ‚Äî use whatever pattern the existing dashboard pages use to get the current user's email.

**Verification**: Visiting `/dashboard/admin/` as a non-owner user redirects to `/dashboard`.

### Task 2: Extend `verifyAdmin` to honour owner emails

The admin API routes currently check `x-admin-api-key` OR `preferences.role === 'admin'`. Owner emails should also pass, since the owner may not have `preferences.role` set.

**Files to update**: `app/api/admin/users/route.ts`, `app/api/admin/audit-log/route.ts`, `app/api/admin/jobs/route.ts`

In each `verifyAdmin` function, after the JWT decode + user lookup, add:

```ts
import { isOwnerEmail } from '@/lib/auth/jwt-utils';
// ...
const user = await prisma.user.findUnique({ where: { id: decoded.userId }, select: { id: true, email: true, preferences: true } });
const prefs = user?.preferences as { role?: string } | null;
if (!user) return { isAdmin: false, error: 'User not found' };
// Owner email bypass
if (isOwnerEmail(user.email)) return { isAdmin: true, userId: decoded.userId };
if (prefs?.role !== 'admin' && prefs?.role !== 'superadmin') {
  return { isAdmin: false, userId: decoded.userId, error: 'Admin access required' };
}
return { isAdmin: true, userId: decoded.userId };
```

Also create a shared `lib/admin/verify-admin.ts` utility to avoid the copy-paste pattern across 3 route files.

**Verification**: Owner user (JWT auth, no API key header) can call `/api/admin/users` and receive `{ success: true, data: [...] }`.

### Task 3: Rewrite `app/dashboard/admin/page.tsx` ‚Äî real data, real actions

Replace the broken Supabase admin client with SWR fetching from `/api/admin/users`.

**Key changes:**
1. Remove `supabase.auth.admin.listUsers()` ‚Äî replace with `useSWR('/api/admin/users?limit=50', fetchJson)` where `fetchJson` uses `credentials: 'include'`
2. `fetchUsers` ‚Üí SWR `mutate()` for refresh
3. `handleUserAction` ‚Üí `fetch('/api/admin/users', { method: 'POST', body: JSON.stringify({ userId, action: mappedAction }) })` where:
   - `'ban'` ‚Üí `'suspend'` (API uses suspend, not ban)
   - `'unban'` ‚Üí `'activate'`
   - `'delete'` ‚Üí `'delete'`
   - `'reset-password'` ‚Üí for now, toast only (no API endpoint yet) ‚Äî log as TODO
4. `handleSaveUser` ‚Üí call `fetch('/api/admin/users', { method: 'POST', ... })` with the edited user's role/status update ‚Äî remove the fake `setTimeout`
5. Stats: Keep `calculateStats()` but feed it from the SWR response `data.data` array instead of Supabase users
6. Keep existing components (`AdminHeader`, `AdminStats`, etc.) ‚Äî they're fine structurally

**SWR pattern** (per CLAUDE.md):
```ts
const { data, isLoading, mutate } = useSWR(
  '/api/admin/users?limit=50',
  (url) => fetch(url, { credentials: 'include' }).then(r => r.json())
);
```

**Verification**: Admin page loads real users from Prisma DB. Suspend action updates user status in DB (verify via audit log or direct DB check).
