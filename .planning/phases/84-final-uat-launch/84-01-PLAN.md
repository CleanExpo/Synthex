# Plan 84-01: Pre-Launch Production Readiness Audit

## objective

Conduct a systematic production readiness audit covering all four quality gates
required before v4.0 is declared Production Complete:

1. **Architecture pattern compliance** — Run the full architecture enforcer
   (P1–P10) against the current codebase and resolve any remaining violations
   discovered after Phase 82 refactoring.

2. **Environment variable audit** — Verify `.env.example` is complete and
   current. Cross-reference every `process.env.*` reference in the codebase
   against the documented variables in `.env.example`. Flag any undocumented
   variables.

3. **Security surface scan** — Check OWASP Top 10 items specific to the
   Next.js 15 / Prisma stack: raw SQL exposure, JWT secret strength, CORS
   policy, CSP headers, rate limiting coverage on public endpoints.

4. **Bundle and performance baseline** — Run `npm run build` and capture the
   production bundle sizes. Flag any route with >200 KB first-load JS or any
   serverless function exceeding the Vercel 50 MB limit.

Linear: SYN-60
Phase dir: `.planning/phases/84-final-uat-launch/`

- `NEXTAUTH_SECRET` / `JWT_SECRET` presence in `.env.example`
- CORS: `middleware.ts` — verify allowed origins list is not `*`
- CSP: `next.config.js` — verify Content-Security-Policy header is set
- Rate limiting: Verify `/api/auth/login`, `/api/auth/register`,
  `/api/auth/request-reset` all have rate limiting applied
- API key exposure: grep for any hardcoded API key patterns (`sk-`, `pk_`)

## tasks

### Task 1: Run Architecture Enforcer (P1–P10)

Run each pattern check from the architecture-enforcer skill.

**P1 — Auth Centralisation:**
```bash
grep -rn "function getJWTSecret\|function getUserFromRequest\|function getUserId(" app/api/ --include="*.ts" | grep -v "lib/auth" | wc -l
# Expected: 0 (Phase 82 resolved all 49 files)
```

**P3 — Icon Barrel Imports:**
```bash
grep -rn "from 'lucide-react'" components/ app/ --include="*.tsx" --include="*.ts" | grep -v "components/icons"
# Expected: 0 results
```

**P4 — Prisma Import Path:**
```bash
grep -rn "new PrismaClient" app/ lib/ --include="*.ts" | grep -v "lib/prisma"
# Expected: 0 results
```

**P7 — Client Directive:**
```bash
grep -rln "useState\|useEffect\|useRef\|useCallback\|useMemo\|useContext" components/ app/ --include="*.tsx" | xargs grep -L "'use client'" 2>/dev/null
# Expected: 0 files
```

**P6 — Dark Theme Tokens (neutral hex only):**
```bash
grep -rn "bg-\[#\|text-\[#\|border-\[#" components/ app/ --include="*.tsx" \
  | grep -v "/80\|/60\|/40\|/20\|/50\|/90\|/95\|/70\|/30\|/10\|/5" \
  | grep -v "#1877F2\|#0A66C2\|#FF0000\|#E60023\|#FF6900\|#BD081C\|#6441A4\|#1DA1F2"
# Expected: 0 results (only platform brand colours remain)
```

**P8 — Fetch Credentials:**
```bash
grep -rn "fetch(" components/ app/ hooks/ --include="*.tsx" --include="*.ts" \
  | grep -v "credentials.*include\|'use server'\|server-only\|node_modules\|api-route\|\.spec\.\|\.test\."
# Review results — flag any client-side fetch missing credentials: 'include'
```

**Action:** Fix any violations found. Document each violation and fix in this task.

### Task 2: Environment Variable Completeness Audit

**Step 1 — Extract all `process.env.` references:**
```bash
grep -rn "process\.env\." app/ lib/ --include="*.ts" --include="*.tsx" \
  | grep -oP "process\.env\.[A-Z_]+" | sort -u
```

**Step 2 — Compare against `.env.example`:**
Extract all documented variable names from `.env.example` and compare against
the Step 1 list. Any variable referenced in code but absent from `.env.example`
must be documented.

**Step 3 — Add missing variables to `.env.example`:**
For each undocumented variable, add it to `.env.example` with:
- A descriptive comment explaining its purpose
- A placeholder value or `# Required` / `# Optional` annotation
- Grouped logically with related variables

**Note:** Do NOT read or modify `.env`, `.env.local`, or `.env.production`.
Only `.env.example` is in scope.

### Task 3: Security Surface Scan

**3a — JWT Secret Strength:**
```bash
grep -rn "JWT_SECRET\|NEXTAUTH_SECRET" .env.example
# Verify these are documented and note minimum length requirement in comment
```

**3b — CORS Policy:**
```bash
grep -n "cors\|CORS\|Access-Control-Allow-Origin" middleware.ts lib/security/ --include="*.ts" -r
# Verify no wildcard '*' origin for credentialed requests
```

**3c — CSP Headers:**
```bash
grep -n "Content-Security-Policy\|contentSecurityPolicy" next.config.js next.config.ts middleware.ts
# Verify CSP header is set in production
```

**3d — Rate Limiting Coverage:**
```bash
grep -rn "rateLimit\|rateLimiter\|rate-limit" app/api/auth/ --include="*.ts"
# Verify login, register, and request-reset endpoints have rate limiting
```

**3e — Hardcoded Secrets:**
```bash
grep -rn "sk-[a-zA-Z0-9]{20,}\|pk_live_\|pk_test_\|whsec_" app/ lib/ --include="*.ts" --include="*.tsx"
# Expected: 0 results — no hardcoded API keys or secrets
```

**3f — Raw SQL Parameterisation:**
```bash
grep -rn "\$queryRaw\|\$executeRaw" lib/ app/ --include="*.ts" | head -20
# Review each result — verify template literal syntax (Prisma.sql) not string concatenation
```

**Action:** Document findings. Fix any critical issues (hardcoded secrets,
wildcard CORS, missing rate limiting on auth routes). Non-critical findings
go into a recommendations section.

### Task 4: Production Build Verification

**Step 1 — Clean build:**
```bash
npm run build 2>&1
```

**Step 2 — Review bundle analysis:**
After the build, capture the First Load JS sizes from Next.js output.
Flag any route that exceeds 200 KB first-load JS.

**Step 3 — Verify build passes without errors:**
- Zero TypeScript errors
- Zero ESLint errors (warnings acceptable)
- All API routes compile successfully
- No `next.config` warnings about deprecated config options

**Step 4 — Serverless function size check:**
Vercel's serverless function limit is 50 MB uncompressed. The `.next/server`
directory contains the compiled functions. Check for any unusually large
function bundles:
```bash
du -sh .next/server/app/api/* 2>/dev/null | sort -rh | head -20
```

**Action:** If build fails, fix the blocking issue. If a bundle exceeds 200 KB,
investigate and recommend code-splitting or lazy loading as a follow-up issue
(do not block launch for this).
