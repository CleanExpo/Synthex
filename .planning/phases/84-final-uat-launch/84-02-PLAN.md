# Plan 84-02: User Acceptance Testing — Critical User Journeys

## objective

Execute UAT across the five critical user journeys that define the v4.0
Production Complete milestone. Each journey is tested end-to-end using the
live dev server (`npm run dev`) against the Supabase development database.

The five journeys are:

1. **Onboarding funnel** — Register → verify email → complete brand profile → connect one social account
2. **Content creation** — Create post → AI generate content → score/preview → schedule → confirm in queue
3. **Cross-platform scheduling** — Schedule a post to 3+ platforms simultaneously → verify queue entries per platform
4. **Admin panel** — Log in as owner → view real users → suspend a test user → verify audit log entry
5. **Subscription and billing** — Navigate to billing → view plan details → verify Stripe customer portal loads

Any journey that fails must be fixed before Phase 84 is marked complete. A
journey that requires a live external API call (Stripe, social platform OAuth)
is marked as smoke-test only — it verifies the UI flow up to the API boundary
and does not require a real transaction.

Linear: SYN-60
Phase dir: `.planning/phases/84-final-uat-launch/`

## context

### UAT Approach

Each journey follows this structure:
1. **Preconditions** — What state the DB and browser must be in before starting
2. **Steps** — Sequential user actions with expected results at each step
3. **Pass criteria** — The specific observable outcome that marks the journey as passing
4. **Known gaps** — Documented limitations (e.g., external API not live in dev)
5. **Fix priority** — If the journey fails: Critical (blocks launch) / High / Medium

### Database State Requirements

Before running UAT:
- At least one test non-owner user exists in the DB (for admin panel journey)
- At least one `PlatformConnection` record exists for the test user (for scheduling journey)
- `Subscription` table has a record for the test user (for billing journey)

If these records do not exist, create them via `npx prisma studio` or seeding
before executing the journey tests.

## tasks

### Task 1: Journey 1 — Onboarding Funnel

**Preconditions:**
- No existing account for the test email
- Email service configured (or email confirmation disabled in Supabase for dev)

**Steps:**
1. Navigate to `/` (landing page) → click "Get Started" or "Sign Up"
2. Complete registration form with a fresh test email
3. If email confirmation required: check confirmation link in email or Supabase dashboard
4. After confirmation: verify redirect to `/onboarding` or `/dashboard`
5. Complete brand profile step (name, industry, website URL)
6. Reach platform connection step (Phase 81 onboarding funnel)
7. Click "Connect" on any platform — verify OAuth redirect initiates (smoke test: does not require completing OAuth)
8. Skip platform connection → verify redirect to dashboard

**Pass criteria:**
- User record created in Prisma `User` table
- `Organization` record created for the user
- Brand profile data persisted in `Organization.settings` or equivalent
- Dashboard loads without errors after onboarding completion
- No JavaScript console errors during the flow

**Known gaps:**
- Email confirmation flow requires dev email service or Supabase email confirmation disabled
- Platform OAuth cannot complete without live OAuth credentials configured

**Fix priority:** Critical — this is the acquisition funnel entry point.

---

### Task 2: Journey 2 — Content Creation Flow

**Preconditions:**
- Logged in as a regular user (not owner)
- `OPENROUTER_API_KEY` set in environment (or mock AI response acceptable)

**Steps:**
1. Navigate to `/dashboard/content` or the content creation page
2. Click "Create Post" or equivalent CTA
3. Select a platform (e.g., Twitter/X)
4. Fill in a topic or keywords
5. Click "Generate with AI" → verify AI generation request fires and returns content
6. Review generated content in editor
7. Observe content score (0–100) displayed in real-time
8. Click "Schedule" → select a date/time
9. Click "Confirm" → verify post appears in queue
10. Navigate to `/dashboard/queue` → verify the post is listed

**Pass criteria:**
- AI generation returns content within 10 seconds (or shows loading state)
- Content score is displayed (any numeric value 0–100)
- Post record created in Prisma `Post` table with status `SCHEDULED`
- Post visible in queue UI with correct scheduled time

**Known gaps:**
- AI generation requires live `OPENROUTER_API_KEY` — if not available, test
  with manual content entry and skip the AI generation sub-step
- AI generation cost is real — use a short test prompt

**Fix priority:** Critical — core product value proposition.

---

### Task 3: Journey 3 — Cross-Platform Scheduling

**Preconditions:**
- Logged in as a regular user
- At least 2 `PlatformConnection` records for the user (created via seed or Prisma Studio)

**Steps:**
1. Navigate to content creation or queue
2. Create a new post with manual content (skip AI generation to save API cost)
3. Select 3 platforms for the post
4. Set scheduled time for 1 hour from now
5. Submit/schedule the post
6. Navigate to queue → verify 3 separate queue entries (one per platform) or a single
   post with 3 `PlatformPost` records

**Pass criteria:**
- `Post` record created in Prisma with status `SCHEDULED`
- Either 3 `PlatformPost` records linked to the post, or 1 post with `platforms` array of 3 entries
- Queue UI shows the post (or 3 entries) with the correct scheduled time
- No 4xx or 5xx errors in browser network tab during submission

**Known gaps:**
- Actual publishing requires live platform credentials — scheduling only is sufficient
- PlatformPost creation logic may depend on Phase 78 queue implementation

**Fix priority:** High — multi-platform scheduling is a key v4.0 differentiator.

---

### Task 4: Journey 4 — Admin Panel

**Preconditions:**
- Logged in as the owner (`phill.mcgurk@gmail.com`)
- At least one non-owner test user exists in the DB

**Steps:**
1. Navigate to `/dashboard/admin`
2. Verify redirect does NOT occur (owner access confirmed)
3. Verify real user list loads from `/api/admin/users` (check network tab)
4. Confirm stats cards show non-zero values (total users, active)
5. Click "Suspend" on a test user → confirm dialog appears → confirm action
6. Verify test user status changes in the UI to "Suspended"
7. Navigate to the Audit Log tab
8. Verify the suspension action appears as an audit log entry with correct metadata
9. Click "Activate" on the same test user → verify status reverts to "Active"
10. Navigate to Platform Health tab → verify job queue stats load without 500 errors

**Pass criteria:**
- Owner can access `/dashboard/admin` without redirect
- Non-owner redirect: log in as a different user and verify `/dashboard/admin` redirects to `/dashboard`
- Suspend action creates an `AuditLog` entry in Prisma
- Audit Log tab shows the entry with correct action, severity, and timestamp
- Platform Health tab loads without errors

**Known gaps:**
- Job queue stats may return empty if BullMQ is not running in dev — empty is acceptable

**Fix priority:** Critical — admin panel is a core operator requirement.

---

### Task 5: Journey 5 — Subscription and Billing (Smoke Test)

**Preconditions:**
- Logged in as a regular user
- `STRIPE_SECRET_KEY` and `STRIPE_PUBLISHABLE_KEY` set (or Stripe test mode)
- Stripe customer record exists for the user (may be auto-created on first visit)

**Steps:**
1. Navigate to `/dashboard/settings` or `/dashboard/billing`
2. Verify current plan is displayed (Free or whichever the test user has)
3. View plan limits (posts/month, platforms, etc.)
4. Click "Manage Billing" or "Upgrade" → verify Stripe customer portal or pricing page loads
5. For upgrade: verify Stripe checkout session creates (network tab shows 200 from `/api/stripe/checkout`)
6. Do NOT complete a real payment — abort at the Stripe checkout page

**Pass criteria:**
- Billing page loads without errors
- Current plan displayed correctly
- Stripe portal/checkout opens (or shows a clear error if Stripe keys are not configured)
- No JavaScript console errors during the flow

**Known gaps:**
- Stripe is on hold per MEMORY.md (UNI-1202/UNI-1203 human-gated)
- If Stripe is not configured, the smoke test passes at the "UI shows plan details" stage
- Real payment flow is explicitly out of scope for this UAT

**Fix priority:** Medium — Stripe integration is human-gated; UI-level pass is sufficient.
