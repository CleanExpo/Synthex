## context

### v4.0 Milestone Definition

v4.0 Production Complete is declared when:
1. All 8 phases (77–84) are complete and committed
2. `npm run release:check` passes (or all failures are pre-existing and documented)
3. All 5 UAT journeys have a Pass or Passed-with-known-gaps status
4. Zero new Critical security findings remain open
5. The codebase is in a committable, deployable state

### What "Production Complete" Means for Synthex

Per MEMORY.md, synthex.social is live but in GOD MODE testing only. v4.0
Production Complete means the codebase is ready for the owner to flip the
switch to public — not that it has been deployed publicly. Deployment is
human-gated (UNI-1182).

### Known Pre-Existing Failures (Excluded from Gate)

These 32 test failures are pre-existing and documented:
- BullMQ transform issues (Jest ESM transform limitation)
- Stripe mock issues (SubscriptionService mock configuration)
- SubscriptionService unit tests (external dependency mocking)

These do NOT block the v4.0 release gate.

## tasks

### Task 1: Resolve Blocking Findings from 84-01 and 84-02

**Triage criteria:**
- **Critical (must fix)**: Security vulnerabilities, data loss risks, complete
  journey failures, type errors blocking build
- **High (should fix)**: Non-critical UAT failures, performance issues, partial
  journey failures with workaround
- **Medium/Low (log as issue)**: Cosmetic issues, non-blocking UX gaps,
  performance optimisation opportunities

**Process:**
1. Read 84-01 and 84-02 findings (from execution notes)
2. List all Critical items
3. Fix each Critical item in order of blast radius (smallest change first)
4. After each fix: run `npm run type-check && npm run lint` to verify no regressions
5. Run `npm test` after all Critical fixes complete

**For each fix:**
- Note the file changed, the issue, and the resolution
- Include in the Phase 84 SUMMARY commit
