## Tasks

### Task 1: Deploy to production (auto)

1. Check if `MAINTENANCE_MODE` var exists in Vercel production — if so, remove it first:
   ```bash
   vercel env ls --environment=production
   # If MAINTENANCE_MODE present: vercel env rm MAINTENANCE_MODE production
   ```

2. Deploy to production:
   ```bash
   vercel --prod
   ```

3. Capture the deployment URL from output (format: `https://synthex-HASH.vercel.app` + promoted to `https://synthex.social` if domain is configured).

4. Wait for deployment to complete (Vercel CLI shows progress). Do not proceed until CLI confirms "Production: https://synthex.social".

### [CHECKPOINT: human-verify]

**User verifies the deployment is working:**

1. Open https://synthex.social in a browser
2. Confirm the page loads (not a Vercel error page or maintenance page)
3. Confirm signup/login page renders correctly

**Then confirm in chat before proceeding to smoke test step.**

### Task 2: Smoke test + health check (auto)

Run the smoke test against the production URL:
```bash
BASE_URL=https://synthex.social node scripts/smoke-test.mjs
```

Expected: 7/7 passed.

Also check health endpoint:
```bash
curl -s https://synthex.social/api/health | head -c 200
```

Document results. If smoke test shows failures:
- Diagnose via Vercel Functions logs: `vercel logs --prod`
- Check for missing env vars (most common cause)
- Fix before marking phase complete

## Verification

- `vercel --prod` completes without error
- https://synthex.social loads in browser (confirmed by user)
- Smoke test: 7/7 passed (or known-acceptable failures documented)
- `/api/health` returns 200 or 503 (503 is acceptable — means DB connected but external services not yet configured)
