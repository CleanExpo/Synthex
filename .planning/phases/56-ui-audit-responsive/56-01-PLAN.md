<task type="auto">
  <name>Task 1: Audit and fix touch target sizes in dashboard header</name>
  <files>
    app/dashboard/layout.tsx
    components/ui/button.tsx
  </files>
  <action>
    Read app/dashboard/layout.tsx in full. Identify all interactive elements in the header and mobile menu toggle:
    - Look for buttons using size="icon-sm" (32×32px) or size="sm" (36px height) on mobile
    - Look for any icon-only buttons without adequate size

    WCAG 2.5.5 requires 44×44px minimum touch targets (or 24px with adequate spacing).
    The coarse-pointer media query in globals.css sets min-height/min-width: 48px for buttons,
    but this only applies at runtime — the declared size classes still control layout.

    Fix: For any icon-only button in the header that uses size="icon-sm":
    - Change to size="icon" (40×40px) if it's in the header/nav area
    - OR add className="min-w-[44px] min-h-[44px]" to ensure touch target

    IMPORTANT: Only fix interactive elements in header, sidebar, and mobile menu —
    do NOT change icon-sm buttons inside content cards where size is intentional.
    Do not change the button variant or functionality — only the size attribute.
  </action>
  <verify>npx tsc --noEmit 2>&1 | tail -5</verify>
  <done>All interactive header/nav elements have min 40×40px touch targets. TypeScript clean.</done>
</task>

<task type="auto">
  <name>Task 2: Colour contrast verification for primary UI patterns</name>
  <files>
    app/globals.css
    tailwind.config.cjs
  </files>
  <action>
    Review the primary colour combinations used in the glassmorphic UI:

    1. Read tailwind.config.cjs to find the custom colour tokens (cyan, gray backgrounds)
    2. Read globals.css for CSS custom properties (--background, --foreground, etc.)

    Document the key text/background pairs:
    - Primary text (white #ffffff) on dark background (#0f172a or similar)
    - Cyan text (cyan-400 ~#22d3ee) on dark background
    - Gray text (gray-400 ~#9ca3af) on dark background
    - Button text on cyan button background

    Calculate or verify contrast ratios:
    - WCAG AA requires 4.5:1 for normal text, 3:1 for large text (18px+ or 14px+ bold)
    - Note: white on #0f172a = ~15:1 (passes)
    - Note: #22d3ee on #0f172a = ~8.5:1 (passes)
    - Note: #9ca3af on #0f172a = ~5.6:1 (passes)

    This task is documentation-only — no code changes needed if ratios are acceptable.
    If any ratio fails (below 4.5:1 for normal text), document the issue with suggested fix.
  </action>
  <verify>Manual calculation — no automated check needed for this task</verify>
  <done>Colour contrast ratios documented for primary UI patterns. Any failures noted with fix.</done>
</task>

<task type="auto">
  <name>Task 3: Verify mobile menu keyboard accessibility</name>
  <files>
    app/dashboard/layout.tsx
    components/dashboard/MobileMenu.tsx (or similar path)
  </files>
  <action>
    Find and read the MobileMenu component. Check:
    1. Does the mobile menu overlay trap focus when open? (WCAG 2.1.2 — No Keyboard Trap exception: focus should be trapped inside modal-style overlays but must have an escape mechanism)
    2. Does it close on Escape key? (Standard UX pattern, reduces keyboard trap risk)
    3. Does the open/close button have aria-label and aria-expanded?
    4. Does the menu use role="dialog" or role="navigation"?

    If the menu is missing focus trap (items are reachable but background content also focusable):
    - This is a warning, not a blocker — document it but do not implement focus trap in this phase
    - The mobile menu is hidden on desktop (lg:hidden) so keyboard trap only affects mobile users

    Document findings. If aria-label or aria-expanded is missing on the toggle button, add it.
  </action>
  <verify>npx tsc --noEmit 2>&1 | tail -5</verify>
  <done>Mobile menu keyboard accessibility documented. Toggle button has aria-label and aria-expanded.</done>
