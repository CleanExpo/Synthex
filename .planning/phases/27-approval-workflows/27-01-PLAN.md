Add two models to Prisma schema after TeamNotification model:

**ApprovalRequest model:**
```prisma
model ApprovalRequest {
  id            String   @id @default(cuid())

  // Content reference
  contentId     String   // ID of Post, Campaign, etc.
  contentType   String   // 'post', 'campaign', 'media', 'template'

  // Workflow
  workflowId    String?  // Optional WorkflowTemplate reference

  // Submitter
  submittedBy   String   // User ID who submitted
  submitter     User     @relation("SubmittedApprovals", fields: [submittedBy], references: [id], onDelete: Cascade)

  // Status
  status        String   @default("pending") // 'pending', 'in_review', 'approved', 'rejected', 'revision_requested'
  priority      String   @default("normal")  // 'low', 'normal', 'high', 'urgent'

  // Progress
  currentStep   Int      @default(0)
  totalSteps    Int      @default(1)

  // Steps stored as JSON array (matches existing service structure)
  steps         Json     @default("[]")

  // Metadata
  title         String
  description   String?
  dueDate       DateTime?
  metadata      Json?    // platforms, scheduledTime, tags, etc.

  // Organization scope
  organizationId String?
  organization   Organization? @relation(fields: [organizationId], references: [id], onDelete: SetNull)

  // Timestamps
  createdAt     DateTime @default(now())
  updatedAt     DateTime @updatedAt

  @@index([submittedBy])
  @@index([organizationId])
  @@index([status])
  @@index([contentId])
  @@map("approval_requests")
}
```

**WorkflowTemplate model:**
```prisma
model WorkflowTemplate {
  id             String   @id @default(cuid())

  // Organization scope
  organizationId String
  organization   Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)

  // Template info
  name           String
  description    String?

  // Steps stored as JSON array
  steps          Json     @default("[]")

  // Configuration
  isDefault      Boolean  @default(false)
  contentTypes   Json     @default("[\"post\"]") // Array of content types this applies to

  // Timestamps
  createdAt      DateTime @default(now())
  updatedAt      DateTime @updatedAt

  @@index([organizationId])
  @@index([isDefault])
  @@map("workflow_templates")
}
```

Create new file with GET and POST handlers for approval requests.

**GET /api/approvals** - List approval requests for current user
- Use getUserIdFromCookies for auth (from lib/auth/session)
- Query params: status, assignedToMe, submittedByMe, contentType, priority, limit, offset
- For assignedToMe filter: check if userId is in any step's assignedTo array (post-process JSON)
- Return { success: true, data: ApprovalRequest[], total: number }

**POST /api/approvals** - Create new approval request
- Required fields: contentId, contentType, title
- Optional fields: workflowId, priority, dueDate, description, metadata
- If workflowId provided, fetch WorkflowTemplate and create steps from template
- Otherwise create default 2-step workflow: "Initial Review" → "Final Approval" (from existing service)
- Steps structure (JSON array):
  ```typescript
  interface ApprovalStep {
    id: string;
    order: number;
    type: 'review' | 'approval' | 'legal_check' | 'brand_check' | 'final_approval';

    currentApprovals: number;
    isOptional: boolean;
    createdAt: string;
    approvedBy?: string;
    approvedAt?: string;
  }
  ```
- Return { success: true, data: ApprovalRequest }

**Use Zod for validation.** Follow existing pattern from app/api/webhooks/user/route.ts.

<task type="auto">
  <name>Task 3: Create approval request detail API route</name>
  <files>app/api/approvals/[id]/route.ts</files>
  <action>
Create new file with GET, PATCH, DELETE handlers for single approval request.

**GET /api/approvals/[id]** - Get single approval request
- Verify user is submitter OR is assigned to a step OR is in same organization
- Return { success: true, data: ApprovalRequest }

**PATCH /api/approvals/[id]** - Update approval request (approve/reject/revision/comment)
- Accept action field: 'approve', 'reject', 'request_revision', 'resubmit', 'add_comment'
- For 'approve':
  - Verify user is assigned to current step
  - Increment currentApprovals on step
  - If currentApprovals >= requiredApprovals, mark step approved, advance to next step
  - If final step approved, set request status to 'approved'
  - Create TeamNotification for submitter and next step assignees
- For 'reject':
  - Require comment/reason
  - Set request status to 'rejected'
  - Create TeamNotification for submitter
- For 'request_revision':
  - Require comment/feedback
  - Set request status to 'revision_requested'
  - Create TeamNotification for submitter
- For 'resubmit':
  - Verify user is submitter
  - Reset current step status to 'pending'
  - Set request status to 'in_review'
  - Create TeamNotification for step assignees
- For 'add_comment':
  - Add comment to current step's comments array
  - Create TeamNotification for mentioned users (if any)
- Return { success: true, data: ApprovalRequest }

**DELETE /api/approvals/[id]** - Delete/cancel approval request
- Only submitter or org admin can delete
- Return { success: true, message: 'Approval request deleted' }
