  <name>Task 1: Fix auth stubs and disable debug endpoints</name>
  <files>app/api/auth/verify-token/route.ts, app/api/sentry-test/route.ts</files>
  <action>
1. **app/api/auth/verify-token/route.ts** — Currently returns `{ valid: true }` stub regardless of input. Implement properly:
   - Extract JWT from Authorization header (Bearer token) or auth-token cookie
   - Call verifyTokenSafe() from lib/auth/jwt-utils.ts
   - If valid: return `{ valid: true, userId: payload.sub }` (don't expose full payload)
   - If invalid/missing: return `{ valid: false }` with 401 status
   - Support both GET (token in cookie) and POST (token in body or header)
   - Reference: lib/auth/jwt-utils.ts for verifyTokenSafe() signature

2. **app/api/sentry-test/route.ts** — Debug endpoint that should not be accessible in production:
   - Add guard at the top: if NODE_ENV is 'production', return 404 with `{ error: 'Not found' }`
   - Keep existing behavior for development/test environments
   - Don't delete the file — it's useful for dev/staging Sentry verification

Don't: Return full JWT payload from verify-token — only return userId and validity to minimize data exposure.
Don't: Use getServerSession for verify-token — it's meant to be a lightweight token check, not a full session load.
  </action>

<task type="auto">
  <name>Task 2: Audit all API routes for auth coverage</name>
  <files>app/api/**/route.ts</files>
  <action>
1. **Grep all route.ts files** under app/api/ for auth patterns:
   - getUserIdFromCookies, getUserIdFromRequest, getUserIdFromRequestOrCookies
   - verifyToken, verifyTokenSafe, requireAuth, withAuth
   - verifyAdmin, ADMIN_API_KEY
   - CRON_SECRET
   - APISecurityChecker
   - supabase.auth.getUser
   - getServerSession

2. **Categorize every route** into:
   - PROTECTED: Has auth check (any pattern above)
   - PUBLIC-INTENTIONAL: Health checks, auth flow (login/signup/callback), webhooks with signature verification
   - UNPROTECTED-SENSITIVE: Accesses user data, performs mutations, or handles billing WITHOUT auth check

3. **For each UNPROTECTED-SENSITIVE route found:**
   - Add auth check using getUserIdFromCookies() (the primary pattern)
   - Return 401 `{ error: 'Unauthorized' }` if no user
   - If the route needs both cookie and header auth (e.g., called by external services), use getUserIdFromRequestOrCookies()

4. **Document the complete audit** in the SUMMARY.md with a table:
   | Route | Category | Auth Method | Action Taken |

**Routes that SHOULD be public (do not add auth):**
- /api/health/* — Load balancer probes
- /api/auth/login, /signup, /logout — Auth flow
- /api/auth/oauth/* — OAuth initiation and callbacks
- /api/auth/refresh — Token refresh (validates token internally)
- /api/auth/reset, /request-reset, /verify-email — Pre-auth recovery flows
- /api/auth/verify-token — Token validation endpoint
- /api/webhooks/stripe — Uses signature verification instead
- /api/cron/* — Uses CRON_SECRET bearer token

Don't: Add auth to webhook routes — they use signature verification which is the correct pattern.
Don't: Add auth to cron routes — they use CRON_SECRET which is the correct pattern.
Don't: Change existing auth patterns in routes that already have them — only add auth where missing.
Don't: Break any existing route behavior — the auth check should be the FIRST thing in the handler, before existing logic.
  </action>
