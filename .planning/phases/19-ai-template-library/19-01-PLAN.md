Add PromptTemplate model to schema following ContentLibrary pattern:

```prisma
model PromptTemplate {
  id              String   @id @default(cuid())
  userId          String?  // Null for system templates
  organizationId  String?  // For team-shared templates

  name            String
  description     String?
  icon            String?  // Emoji icon
  category        String   // 'marketing', 'engagement', 'educational', 'promotional', 'personal'
  platforms       String[] // ['twitter', 'linkedin', 'instagram', etc.]

  // Template structure (JSON matching ContentTemplate interface)
  structure       Json     // { hook, body, cta?, hashtags?, mediaType? }
  variables       String[] // Placeholder names to fill in
  tips            String[] // Usage tips

  // Visibility & sharing
  isPublic        Boolean  @default(false)  // Available to all users
  isSystem        Boolean  @default(false)  // Built-in template

  // Usage tracking
  usageCount      Int      @default(0)
  lastUsedAt      DateTime?

  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt

  // Relations
  user            User?    @relation(fields: [userId], references: [id], onDelete: Cascade)
  organization    Organization? @relation(fields: [organizationId], references: [id], onDelete: SetNull)

  @@index([userId])
  @@index([category])
  @@index([isPublic])
  @@index([isSystem])
  @@map("prompt_templates")
}
```

Add relation to User model: `promptTemplates PromptTemplate[]`
Add relation to Organization model: `promptTemplates PromptTemplate[]`

Run `npx prisma validate` then `npx prisma generate`.

WHY this structure:
- userId nullable allows system templates (isSystem=true)
- organizationId enables team sharing
- structure as Json matches existing ContentTemplate interface
- isPublic/isSystem separate user-shared vs built-in templates
- usageCount/lastUsedAt tracks popularity (replaces localStorage)

**scripts/seed-templates.ts**:
```typescript
import { PrismaClient } from '@prisma/client';
import { contentTemplates } from '../lib/content-templates';

const prisma = new PrismaClient();

async function seedTemplates() {
  console.log('Seeding system templates...');

  for (const template of contentTemplates) {
    await prisma.promptTemplate.upsert({
      where: {
        // Use unique constraint or check by name + isSystem
        id: `system-${template.id}`,
      },
      update: {
        name: template.name,
        description: template.description,
        icon: template.icon,
        category: template.category,
        platforms: template.platforms,
        structure: template.structure,
        variables: template.variables || [],
        tips: template.tips || [],
        usageCount: template.popularity,
      },
      create: {
        id: `system-${template.id}`,
        userId: null, // System template
        name: template.name,
        description: template.description,
        icon: template.icon,
        category: template.category,
        platforms: template.platforms,
        structure: template.structure,
        variables: template.variables || [],
        tips: template.tips || [],
        isPublic: true,
        isSystem: true,
        usageCount: template.popularity,
      },
    });
  }

  console.log(`Seeded ${contentTemplates.length} system templates`);
}

seedTemplates()
  .catch(console.error)
  .finally(() => prisma.$disconnect());
```

Add to package.json scripts:
```json
"seed:templates": "npx tsx scripts/seed-templates.ts"
```

Update lib/content-templates.ts to add comment noting templates are now in database.

WHY this approach:
- upsert prevents duplicates on re-runs
- system- prefix clearly identifies built-in templates
- Preserves popularity counts as initial usageCount
- Script is idempotent (safe to run multiple times)
