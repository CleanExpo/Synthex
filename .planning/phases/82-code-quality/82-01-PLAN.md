# Plan 82-01: Auth Centralisation — Remove Local Auth Helpers from API Routes

## objective

Remove 49 duplicated local auth helper functions (`getUserFromRequest`, `getUserId`, `getTokenFromRequest`) from API routes and replace them with the canonical `getUserIdFromRequestOrCookies` or `getUserIdFromCookies` from `@/lib/auth/jwt-utils`. This enforces the P1 architecture pattern and eliminates divergent auth logic that is a security maintenance risk.

Linear: SYN-57
Phase dir: `.planning/phases/82-code-quality/`

## execution_context

- Canonical auth exports: `lib/auth/jwt-utils.ts`
  - `getUserIdFromCookies()` — async, reads `auth-token` cookie, returns `string | null`
  - `getUserIdFromRequestOrCookies(request: NextRequest)` — checks header then cookie
  - `verifyToken(token: string)` — raw JWT verification, returns `JWTPayload`
  - `unauthorizedResponse()` — returns `NextResponse` 401

## context

### Discovery Findings

**Pattern A — Thin cookie wrapper (28 files)**:
These define `async function getUserId(_request: NextRequest): Promise<string | null>` that just calls `getUserIdFromCookies()`. The `request` parameter is unused. The local function is pure boilerplate.

Example from `app/api/ab-testing/tests/route.ts`:
```ts
async function getUserId(_request: NextRequest): Promise<string | null> {
  return getUserIdFromCookies();
}
```

Fix: Delete the local function, import `getUserIdFromCookies` from `@/lib/auth/jwt-utils`, call it directly.

**Pattern B — Bearer + cookie dual-check (21 files)**:
These define `async function getUserFromRequest(request: NextRequest)` that checks `Authorization: Bearer <token>` header first, then falls back to `auth-token` cookie. This matches exactly what `getUserIdFromRequestOrCookies` does.

Example from `app/api/affiliates/links/route.ts`:
```ts
async function getUserFromRequest(request: NextRequest): Promise<{ id: string } | null> {
  const authHeader = request.headers.get('authorization');
  if (authHeader) {
    const token = authHeader.replace('Bearer ', '');
    const decoded = verifyToken(token);
    return { id: decoded.userId };
  }
  const authToken = request.cookies.get('auth-token')?.value;
  if (authToken) {
    const decoded = verifyToken(authToken);
    return { id: decoded.userId };
  }
  return null;
}
```

Fix: Delete the local function. Replace call sites with `getUserIdFromRequestOrCookies(request)` which returns `string | null`. Update call sites that expect `{ id: string } | null` to work with `string | null`.

### Impact Assessment

- No behaviour change: The replacements are functionally identical to the originals
- Return type changes: Pattern B returns `{ id: string } | null`, canonical returns `string | null`. Call sites using `userId?.id` must change to just `userId`.
- Zero Prisma schema changes required
- Zero API contract changes (same auth semantics)
