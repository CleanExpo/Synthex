## context

### Why console.error → logger.error Matters

The project uses `logger` from `lib/logger.ts` which routes to Sentry in production (see Phase 73 security hardening). Files using `console.error` instead bypass:
- Structured log levels (configurable per environment)
- Sentry error capture for unexpected errors
- Consistent log format across the platform
- Production log aggregation

The migration is mechanical: `console.error('message', error)` → `logger.error('message', { error })`. The signature differs slightly — `logger` accepts an optional metadata object as the second argument.

### Batch Strategy — Logger Migration

With 531 occurrences across ~180 files, process in route groups:
- Group A: `app/api/ab-testing/` (6 files, ~12 calls)
- Group B: `app/api/affiliates/` (5 files, ~10 calls)
- Group C: `app/api/analytics/` through `app/api/content/` (~30 files)
- Group D: `app/api/integrations/` through `app/api/reporting/` (~40 files)
- Group E: `app/api/revenue/` through `app/api/unified/` (~50 files)

### Icon Barrel — Direct Imports

`FileQuestion` is used in both not-found pages but is NOT currently exported from `components/icons.tsx`.

Options:
1. Add `FileQuestion` to `components/icons.tsx` barrel export and update the 2 files — **preferred**
2. Leave the 2 files as-is and document the exception

**Decision**: Option 1. Add `FileQuestion` to the barrel export. This is a low-risk additive change and keeps the P3 pattern clean.
