  <files>

  <action>
Create lib/seo/pagespeed-service.ts with three exported functions:

1. `runPageSpeedAnalysis(url: string, strategy: 'mobile' | 'desktop')` — Calls PageSpeed Insights API v5 (https://www.googleapis.com/pagespeedonline/v5/runPagespeed) with categories PERFORMANCE, ACCESSIBILITY, SEO, BEST_PRACTICES. Extract:
   - Lighthouse category scores (performance, seo, accessibility, best-practices) as 0-100
   - Field data (CrUX): LCP, CLS, INP, FID from loadingExperience.metrics (with percentile values)
   - Lab data: LCP, CLS, TBT, Speed Index, FCP from lighthouseResult.audits
   - Top 5 opportunities (from lighthouseResult.audits where details.type === 'opportunity') with title, description, savings
   - Top 5 diagnostics (from lighthouseResult.audits where score !== null && score < 1) with title, description, displayValue
   - Use GOOGLE_PAGESPEED_API_KEY env var if present (optional, for higher rate limits)
   - If API fails or credentials missing, return graceful demo data with isDemo: true flag (same pattern as search-console.ts)
   - 30s timeout via AbortSignal.timeout(30000)

2. `getPageSpeedHistory(userId: string, limit?: number)` — Query SEOAudit records via Prisma where userId matches, ordered by createdAt desc, limit default 20. Map to PageSpeedHistoryEntry type with url, date, performanceScore, seoScore, accessibilityScore, bestPracticesScore, lcp, cls, inp.

3. `getPerformanceTrends(userId: string, days?: number)` — Query SEOAudit records for last N days (default 30), group by date, calculate daily averages for performance score, LCP, CLS, INP. Return array of { date, avgPerformance, avgLcp, avgCls, avgInp }.

Types to export: PageSpeedAnalysisResult, PageSpeedOpportunity, PageSpeedDiagnostic, PageSpeedHistoryEntry, PerformanceTrendPoint, FieldMetrics, LabMetrics.

Create 3 API routes following exact pattern from cwv-history/route.ts:
- POST /api/seo/pagespeed/analyze — accepts { url: string, strategy?: 'mobile' | 'desktop' } via Zod. Calls runPageSpeedAnalysis. Returns full analysis result.
- GET /api/seo/pagespeed/history — calls getPageSpeedHistory(userId). Returns { success, history, total }.
- GET /api/seo/pagespeed/trends — accepts optional ?days=30 query param. Calls getPerformanceTrends(userId, days). Returns { success, trends }.

All routes: APISecurityChecker with DEFAULT_POLICIES.AUTHENTICATED_READ (GET) or DEFAULT_POLICIES.AUTHENTICATED_WRITE (POST), subscription check (plan !== 'free' → 402), proper error handling with console.error and 500 response.

Do NOT duplicate fetchPageSpeedData from technical-seo-service.ts — write a new expanded version in pagespeed-service.ts that extracts more data (opportunities, diagnostics, field vs lab distinction). The old one remains for mobile parity only.
