<task type="auto">
  <name>Task 2: Create RBAC permission engine test suite</name>
  <files>tests/unit/lib/auth/rbac.test.ts</files>
  <action>
Create tests for the RBAC system: permission-engine.ts, role-manager.ts, and access-control.ts.

**Setup:**
- Mock `@/lib/prisma` for all RBAC database operations
- Import RBAC facade from lib/auth/rbac/index.ts or individual modules

**Test categories:**

1. **Permission matching** (access-control.ts):
   - Exact match: user has 'posts:create' → check 'posts:create' → true
   - Wildcard: user has 'posts:*' → check 'posts:create' → true
   - Super admin: user has '*' → check anything → true
   - Resource-scoped: user has 'posts:read:own' → check with ownership context
   - No match: user has 'posts:read' → check 'posts:create' → false

2. **Role aggregation** (role-manager.ts):
   - Single role permissions resolved correctly
   - Multiple roles: permissions merged (union)
   - Role hierarchy: admin inherits member permissions
   - Empty roles: no permissions granted
   - Create/delete role operations with Prisma mocks

3. **Permission engine** (permission-engine.ts):
   - getUserPermissions() returns aggregated permissions from all roles
   - check() integrates role lookup + permission matching
   - Caching: second call uses cache (verify Prisma not called twice)
   - Cache invalidation on role change

4. **Edge cases:**
   - User with no roles → no permissions
   - Deleted role → permissions removed
   - Invalid permission format → handled gracefully
   - Concurrent permission checks → no race conditions

**Mocking strategy:**
- Prisma: Mock userRole.findMany, role.findUnique, etc.
- Return predefined role/permission fixtures
- Use jest.spyOn for cache verification

**Test count target:** 20-25 test cases across all RBAC components.

Don't: Test UI authorization components — only lib/auth/rbac/ logic.
Don't: Create integration tests against real database.
Reference: Read each RBAC file first to understand the exact API surface and method signatures.
