# Plan 79-01: Role Update API & Edit-User Dialog Wiring

## objective

Add a `PATCH /api/admin/users` endpoint that updates a user's role and status via Prisma, then wire the existing `EditUserDialog` to call it. Currently the dialog has working Role and Status dropdowns but the `handleSaveUser` function only maps status changes to suspend/activate/delete actions -- it cannot update a user's role (e.g. promoting a user to admin or moderator). This plan adds the missing PATCH handler and updates the dialog to send both role and status in a single request.

Also wire the admin-triggered password reset action. The admin page shows "Password reset emails are not yet supported via the API" when the reset-password action is clicked, but `/api/auth/request-reset` already exists and accepts `{ email }`. This plan connects the two.

Linear: SYN-18
Phase dir: `.planning/phases/79-admin-completion/`

## execution_context

- Admin users API: `app/api/admin/users/route.ts` -- GET (list) + POST (suspend/activate/delete), no PATCH
- Edit dialog: `components/admin/edit-user-dialog.tsx` -- Role + Status selects, calls `onSave()`
- Admin page: `app/dashboard/admin/page.tsx` -- `handleSaveUser` maps to POST with action only, no role update
- Admin types: `components/admin/types.ts` -- `User` type has `role` and `status` fields

- Password reset API: `app/api/auth/request-reset/route.ts` -- POST `{ email }` sends reset email via `emailService`
- Shared admin auth: `lib/admin/verify-admin.ts` -- used by all admin routes

## context

### Discovery Findings

**Gap 1 -- No role update**: The POST handler in `/api/admin/users` only accepts `action: 'suspend' | 'activate' | 'delete'`. It updates `preferences.status` but has no mechanism to update `preferences.role`. The `EditUserDialog` has a Role dropdown (user/admin/moderator) but changes are silently lost because `handleSaveUser` in the page only sends the status-derived action.

**Gap 2 -- Email change ignored**: The dialog has an email input field, but `handleSaveUser` never sends the email value. Email changes should either be sent to the API or the field should be read-only. Given the security implications of email changes (breaks auth, invalidates tokens), the field should be made read-only.

**Gap 3 -- Password reset disconnect**: `handleUserAction('reset-password')` shows a toast instead of calling the existing `/api/auth/request-reset` endpoint with the user's email.

### Architecture Decision

- Add `PATCH /api/admin/users` handler that accepts `{ userId, role?, status? }` and updates `preferences` JSON field accordingly
- Keep POST for destructive actions (suspend/activate/delete) -- these involve transactions with session invalidation and audit logs
- PATCH is for non-destructive metadata updates (role change, status change without session invalidation)
- Wire `handleSaveUser` to call PATCH with the edited role and status
- Wire `handleUserAction('reset-password')` to call `/api/auth/request-reset` with the target user's email
- Make email field read-only in the edit dialog (email changes have cascading auth implications)

## tasks

### Task 1: Add PATCH handler to `app/api/admin/users/route.ts`

Add a PATCH export to the existing admin users route.

**Zod schema:**
```ts
const updateUserSchema = z.object({
  userId: z.string().uuid(),
  role: z.enum(['user', 'admin', 'moderator']).optional(),
  status: z.enum(['active', 'suspended', 'banned']).optional(),
});
```

**Logic:**
1. Verify admin via `verifyAdmin(request)`
2. Validate request body
3. Look up target user -- reject if target is superadmin (cannot downgrade superadmins)
4. Enforce org scope (same as POST handler): JWT-authenticated admins cannot modify users from other orgs
5. Merge role/status into existing `preferences` JSON:
   ```ts
   const currentPrefs = (targetUser.preferences as Record<string, unknown>) || {};
   const updatedPrefs = { ...currentPrefs };
   if (role) updatedPrefs.role = role;
   if (status) updatedPrefs.status = status;
   ```
6. Update user via Prisma in a transaction with audit log entry
7. If status changed to `suspended` or `banned`, also delete sessions (same as POST suspend handler)
8. Return `{ success: true, message: 'User updated' }`

**Audit log entry:**
```ts
await tx.auditLog.create({
  data: {
    userId: auth.userId || 'system',
    action: 'user_role_updated',
    resource: 'user',
    resourceId: targetUser.id,
    details: {
      targetEmail: targetUser.email,
      changes: { role, status },
      previousRole: currentPrefs.role,
      previousStatus: currentPrefs.status,
    },
    severity: role === 'admin' ? 'high' : 'medium',
    category: 'admin',
    outcome: 'success',
  },
});
```

### Task 2: Update `handleSaveUser` in `app/dashboard/admin/page.tsx`

Replace the current POST-based save with a PATCH call:

```ts
const handleSaveUser = useCallback(async () => {
  if (!editingUser) return;
  setIsSaving(true);
  try {
    const res = await fetch('/api/admin/users', {
      method: 'PATCH',
      credentials: 'include',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        userId: editingUser.id,
        role: editingUser.role || 'user',
        status: editingUser.status || 'active',
      }),
    });
    const json = await res.json();
    if (!res.ok) throw new Error(json.message ?? 'Failed to update user');
    toast.success('User updated successfully');
    setEditDialogOpen(false);
    setEditingUser(null);
    mutate();
  } catch (error) {
    toast.error(error instanceof Error ? error.message : 'Failed to update user');
  } finally {
    setIsSaving(false);
  }
}, [editingUser, mutate]);
```

### Task 3: Wire password reset action

In `handleUserAction` (case `'reset-password'`), replace the toast-only stub with a real API call:

```ts
case 'reset-password': {
  // Find the user's email from the loaded users array
  const targetUser = users.find(u => u.id === userId);
  if (!targetUser?.email) {
    toast.error('User email not found');
    break;
  }
  try {
    const res = await fetch('/api/auth/request-reset', {
      method: 'POST',
      credentials: 'include',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email: targetUser.email }),
    });
    if (!res.ok) throw new Error('Failed to send reset email');
    toast.success(`Password reset email sent to ${targetUser.email}`);
  } catch (error) {
    toast.error(error instanceof Error ? error.message : 'Failed to send password reset');
  }
  break;
}
```

Update the `handleUserAction` dependency array to include `users`.
