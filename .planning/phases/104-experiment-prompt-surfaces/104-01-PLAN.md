## Summary

Wire Bayesian Optimisation into two existing Growth-tier surfaces:

1. **`experiment_sampling`** — BO-optimised sampling ratios for the A/B
   experiment designer (`lib/experiments/experiment-designer.ts`). The surface
   controls four variant-allocation parameters: the probability weight applied
   to each of the four highest-priority experiment types when the designer
   selects which type to suggest first for a given URL.

2. **`prompt_testing`** — BO-optimised parameters for the prompt tester
   (`lib/prompts/prompt-tester.ts`). The surface controls three model
   parameters: `temperature` (0.3–0.9), `maxTokens` (400–1200 in 100-unit
   steps mapped as a float 0.4–1.2 normalised), and a `positivityBias` weight
   (0.0–1.0) that adjusts how the response-quality scorer weights positive
   quality markers vs. structural signals.

Both surfaces follow the established adapter pattern from
`lib/bayesian/surfaces/geo-weights.ts` and `lib/bayesian/surfaces/tactic-weights.ts`.
Neither surface requires schema changes, new API routes, or UI component additions
beyond the BO dashboard already listing both surfaces by name from Phase 103.

**User-visible changes (Growth+ plan only):**
- Experiment designer returns suggestions in a BO-informed priority order instead
  of a fixed rule order.
- Prompt tester uses BO-optimised `temperature` and quality-scoring parameters
  instead of fixed constants.
- The BO dashboard at `/dashboard/optimisation` will automatically show
  `experiment_sampling` and `prompt_testing` spaces once observations are recorded.

## Surface Design

### Surface 1 — `experiment_sampling`

**What it optimises:** The probability distribution applied when selecting which
experiment type to suggest as highest-priority for a URL. Currently,
`suggestExperiments()` applies all applicable rules and sorts high > medium > low
using fixed priority values embedded in each rule definition.

With BO enabled, the adapter provides a `samplingWeights` object with one weight
per experiment type. The designer uses these weights to re-score rules — a rule
whose type has a higher weight gets boosted to the front.

**Dimensions (6):**

| Parameter | Default | Min | Max | Rationale |
|-----------|---------|-----|-----|-----------|
| `titleTag` | 0.20 | 0.05 | 0.40 | Weight for title-tag suggestions |
| `metaDescription` | 0.20 | 0.05 | 0.40 | Weight for meta-description suggestions |
| `h1` | 0.20 | 0.05 | 0.40 | Weight for h1 suggestions |
| `schema` | 0.15 | 0.05 | 0.40 | Weight for schema suggestions |
| `contentStructure` | 0.15 | 0.05 | 0.40 | Weight for content-structure suggestions |
| `internalLinks` | 0.10 | 0.05 | 0.40 | Weight for internal-links suggestions |

Sum is not constrained to 1 because weights are used as relative priority scores,
not probability distributions. No `sumEquals` constraint is set.

**Fallback behaviour:** When BO is unavailable, the adapter returns the default
weights above, which reproduce the existing fixed-priority order (all `high`
priority rules use the same priority, but title/meta/h1 map to slightly higher
defaults, reflecting their current `high` vs `medium` designation).

**Target metric for BO observations:** `improvement` from `ExperimentResult`
(the `improvement` field on completed `SEOExperiment` records — the percentage
score improvement of the winning variant vs baseline).

**Integration point in `experiment-designer.ts`:** The `suggestExperiments()`
function currently sorts by a fixed `priorityOrder` after building the suggestions
array. Modify this to accept an optional `samplingWeights` argument and use it to
re-order suggestions. The caller (the API route at
`app/api/experiments/experiments/route.ts`) needs to fetch sampling weights and
pass them in before calling `suggestExperiments()`.

### Surface 2 — `prompt_testing`

**What it optimises:** Three numeric parameters in the prompt test call:

| Parameter | Default | Min | Max | Rationale |
|-----------|---------|-----|-----|-----------|
| `temperature` | 0.7 | 0.30 | 0.90 | LLM temperature — lower = more deterministic |
| `maxTokens` | 800 | 400 | 1200 | Max response tokens — affects length and detail |
| `positivityBias` | 0.5 | 0.0 | 1.0 | Scalar applied to positive marker bonus in quality scoring |

`temperature` and `maxTokens` are passed directly to the OpenRouter API call.
`positivityBias` is used in `scoreResponseQuality()` to scale the per-hit
bonus for positive quality markers (`positiveHits * 0.03 * positivityBias`
instead of the fixed `0.03`).

**Target metric for BO observations:** `responseQuality` from `PromptTestResult`
(the `responseQuality` float 0–1 score already saved to `PromptResult` in the DB).
When `brandMentioned === true`, the quality value is the direct observation target.
When `brandMentioned === false`, target is `responseQuality * 0.5` (penalise
responses that don't mention the brand at all).

**Fallback behaviour:** Returns the current hardcoded constants:
`{ temperature: 0.7, maxTokens: 800, positivityBias: 0.5 }`.

## Architecture

### Sampling Weights Integration Pattern

```
suggestExperiments(url, metrics)  [current - no weights]
       ↓
suggestExperiments(url, metrics, samplingWeights?)  [new - optional weights]
       ↓
If samplingWeights provided:
  - For each suggestion, look up its experimentType in samplingWeights
  - Compute effectivePriority = priorityOrder[suggestion.priority] - weight * 10
  - Sort by effectivePriority ascending (lower = higher priority)
Else:
  - Use existing fixed priorityOrder sort (unchanged)
```

This is purely additive. The caller decides whether to fetch BO weights and pass
them. When `samplingWeights` is `undefined`, the function behaves identically to
the current implementation.

### Prompt Testing Integration Pattern

```typescript
// In app/api/prompts/test/route.ts — before calling testPrompt():
const user = await prisma.user.findUnique({
  where: { id: userId },
  select: { organizationId: true, plan: true },

  ? await getPromptTestingParams(orgId)
  : undefined;

const testResult = await testPrompt(tracker.promptText, tracker.entityName, boParams);

// After test: register BO observation
if (boParams?.source === 'bo') {
  const target = testResult.brandMentioned
    ? testResult.responseQuality
    : testResult.responseQuality * 0.5;

  await registerObservationSilently(
    'prompt_testing',
    orgId,
    {
      temperature:    boParams.params.temperature,
      maxTokens:      boParams.params.maxTokens / 1000,  // normalise 0.4–1.2
      positivityBias: boParams.params.positivityBias,
    },
    target,
    { brandMentioned: testResult.brandMentioned, responseQuality: testResult.responseQuality }
  );
}
```

**File**: `lib/bayesian/surfaces/experiment-sampling.ts`

```typescript
/**
 * Bayesian Optimisation Surface Adapter — Experiment Sampling
 *
 * Provides BO-optimised sampling weights for the experiment type priority
 * order in the A/B experiment designer.
 *
 * Surface: 'experiment_sampling'
 * Dimensions: 6 experiment type weights (no sum constraint)
 *
 * @module lib/bayesian/surfaces/experiment-sampling
 */

import { getOptimisedWeightsOrFallback } from '@/lib/bayesian/fallback';
import type { WeightResult } from '@/lib/bayesian/fallback';
import type { ParameterBounds } from '@/lib/bayesian/types';

export interface ExperimentSamplingWeights {
  titleTag:        number;
  metaDescription: number;
  h1:              number;
  schema:          number;
  contentStructure: number;
  internalLinks:   number;
}

export interface ExperimentSamplingResult {
  weights: ExperimentSamplingWeights;
  source:  WeightResult['source'];
}

export const EXPERIMENT_SAMPLING_DEFAULTS: ExperimentSamplingWeights = {
  titleTag:        0.20,
  metaDescription: 0.20,
  h1:              0.20,
  schema:          0.15,
  contentStructure: 0.15,
  internalLinks:   0.10,
};

export const EXPERIMENT_SAMPLING_BOUNDS: Record<keyof ExperimentSamplingWeights, ParameterBounds> = {
  titleTag:        { min: 0.05, max: 0.40 },
  metaDescription: { min: 0.05, max: 0.40 },
  h1:              { min: 0.05, max: 0.40 },
  schema:          { min: 0.05, max: 0.40 },
  contentStructure: { min: 0.05, max: 0.40 },
  internalLinks:   { min: 0.05, max: 0.40 },
};

export async function getExperimentSamplingWeights(
  orgId: string
): Promise<ExperimentSamplingResult> {
  const result = await getOptimisedWeightsOrFallback(
    'experiment_sampling',
    orgId,
    () => ({ ...EXPERIMENT_SAMPLING_DEFAULTS }),
  );

  const raw = result.weights as Record<string, number>;
  const weights: ExperimentSamplingWeights = {
    titleTag:        raw.titleTag        ?? EXPERIMENT_SAMPLING_DEFAULTS.titleTag,
    metaDescription: raw.metaDescription ?? EXPERIMENT_SAMPLING_DEFAULTS.metaDescription,
    h1:              raw.h1              ?? EXPERIMENT_SAMPLING_DEFAULTS.h1,
    schema:          raw.schema          ?? EXPERIMENT_SAMPLING_DEFAULTS.schema,
    contentStructure: raw.contentStructure ?? EXPERIMENT_SAMPLING_DEFAULTS.contentStructure,
    internalLinks:   raw.internalLinks   ?? EXPERIMENT_SAMPLING_DEFAULTS.internalLinks,
  };

  return { weights, source: result.source };
}
```

### Task 2 — Create `lib/bayesian/surfaces/prompt-testing.ts`

**File**: `lib/bayesian/surfaces/prompt-testing.ts`

Create the surface adapter for the prompt testing surface. Note: the file is
named `prompt-testing.ts` (kebab-case) to match directory naming convention.
The surface identifier is `'prompt_testing'` (snake_case) as declared in `BOSurface`.

```typescript
/**
 * Bayesian Optimisation Surface Adapter — Prompt Testing
 *
 * Provides BO-optimised parameters for the AI prompt tester.
 * Controls LLM temperature, max token budget, and quality scoring bias.
 *
 * Surface: 'prompt_testing'
 * Dimensions: temperature, maxTokens (normalised), positivityBias
 *
 * @module lib/bayesian/surfaces/prompt-testing
 */

import { getOptimisedWeightsOrFallback } from '@/lib/bayesian/fallback';
import type { WeightResult } from '@/lib/bayesian/fallback';
import type { ParameterBounds } from '@/lib/bayesian/types';

export interface PromptTestingParams {
  temperature:    number;  // 0.30 – 0.90
  maxTokens:      number;  // 400 – 1200 (integer, BO returns float — round before use)
  positivityBias: number;  // 0.0 – 1.0
}

export interface PromptTestingParamsResult {
  params:  PromptTestingParams;
  source:  WeightResult['source'];
}

export const PROMPT_TESTING_DEFAULTS: PromptTestingParams = {
  temperature:    0.70,
  maxTokens:      800,
  positivityBias: 0.50,
};

export const PROMPT_TESTING_BOUNDS: Record<keyof PromptTestingParams, ParameterBounds> = {
  temperature:    { min: 0.30, max: 0.90 },
  maxTokens:      { min: 400,  max: 1200 },
  positivityBias: { min: 0.00, max: 1.00 },
};

export async function getPromptTestingParams(
  orgId: string
): Promise<PromptTestingParamsResult> {
  const result = await getOptimisedWeightsOrFallback(
    'prompt_testing',
    orgId,
    () => ({
      temperature:    PROMPT_TESTING_DEFAULTS.temperature,
      maxTokens:      PROMPT_TESTING_DEFAULTS.maxTokens,
      positivityBias: PROMPT_TESTING_DEFAULTS.positivityBias,
    }),
  );

  const raw = result.weights as Record<string, number>;
  const params: PromptTestingParams = {
    temperature:    raw.temperature    ?? PROMPT_TESTING_DEFAULTS.temperature,
    maxTokens:      Math.round(raw.maxTokens ?? PROMPT_TESTING_DEFAULTS.maxTokens),
    positivityBias: raw.positivityBias ?? PROMPT_TESTING_DEFAULTS.positivityBias,
  };

  return { params, source: result.source };
}
```

Note: `maxTokens` is rounded because BO returns float suggestions but the OpenRouter
API expects an integer. The `Math.round()` call ensures type safety.

Verification: `npm run type-check` — zero errors expected.

**File**: `lib/experiments/experiment-designer.ts`

  _url: string,
  currentMetrics: CurrentMetrics,
  samplingWeights?: ExperimentSamplingWeights,

The type-to-key mapping from `ExperimentType` to `ExperimentSamplingWeights` keys:

```typescript
const TYPE_TO_WEIGHT_KEY: Record<string, keyof ExperimentSamplingWeights> = {
  'title-tag':        'titleTag',
  'meta-description': 'metaDescription',
  'h1':               'h1',
  'schema':           'schema',
  'content-structure': 'contentStructure',
  'internal-links':   'internalLinks',
};
```

Replace the existing sort:

```typescript
// Before:
const priorityOrder = { high: 0, medium: 1, low: 2 };
return suggestions.sort(
  (a, b) => priorityOrder[a.priority] - priorityOrder[b.priority]
);

// After:
const priorityOrder = { high: 0, medium: 1, low: 2 };
return suggestions.sort((a, b) => {
  // Base sort: fixed priority order (high first)
  const baseDiff = priorityOrder[a.priority] - priorityOrder[b.priority];
  if (!samplingWeights) return baseDiff;

  // BO tie-breaking: within same priority tier, higher weight comes first
  if (baseDiff !== 0) return baseDiff;
  const weightA = samplingWeights[TYPE_TO_WEIGHT_KEY[a.experimentType]] ?? 0;
  const weightB = samplingWeights[TYPE_TO_WEIGHT_KEY[b.experimentType]] ?? 0;
  return weightB - weightA;  // descending weight = ascending sort position
});
```

This is purely additive: when `samplingWeights` is undefined, the sort is
identical to the current implementation. When weights are provided, BO
influences ordering only within equal-priority tiers.

**Do not** modify the caller or add any async behaviour to `suggestExperiments`.
The function remains synchronous — the caller fetches BO weights before invoking it.

Find the actual caller of `suggestExperiments` using Grep before finalising this
task. It may be called from a component or a different API route.

## Risk Notes

**Risk 1 — User.plan field name**
The Prisma `User` model field for the subscription plan may be named
`subscriptionPlan`, `tier`, or similar. Run:
```bash
grep -n "plan\|tier\|subscription" prisma/schema.prisma
```
before implementing Tasks 5 and 6. Adjust field names accordingly.

**Risk 2 — suggestExperiments may be called from a client component**
If `suggestExperiments` is called client-side (e.g. in `ExperimentWizard.tsx`),
the BO weight fetch cannot happen there — it requires server context.
In that case, the weight fetch must be added to the API endpoint and the
weights passed down as props or included in the API response.
Grep for the call site before Task 6.

**Risk 3 — parseResponse is directly tested**
`lib/prompts/prompt-tester.ts` exports `parseResponse` (confirmed — line 94).
If unit tests call `parseResponse` with only 2 arguments, adding the optional
third parameter will not break them (TypeScript optional params are backward
compatible). Verify with `npm test` after Task 4.

**Risk 4 — maxTokens integer rounding**
BO returns float suggestions. `Math.round(raw.maxTokens ?? 800)` in Task 2
handles this. Verify the OpenRouter API rejects non-integer `max_tokens` values
by checking if any test already validates this — if not, note it.

**Risk 5 — Plan value mismatch between DB and feature-limits.ts**
`BO_PLAN_LIMITS` uses `'growth'` as the plan key. The DB may store `'Growth'`
or `'GROWTH'`. `isSurfaceAvailable(plan, surface)` does a direct string lookup
so the case must match. Read the plan normalisation logic in
`lib/bayesian/feature-limits.ts` line 76 — it falls back to `free` for
unrecognised plans. If DB stores mixed case, add `.toLowerCase()` to the plan
string before passing to `isSurfaceAvailable()`.
