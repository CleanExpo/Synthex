## Scale-Tier Gating Pattern

Every integration point follows this pattern:

```typescript
import { isSurfaceAvailable } from '@/lib/bayesian/feature-limits';
import { get<SurfaceName>Weights } from '@/lib/bayesian/surfaces/<surface-name>';

// Resolve plan — fallback to 'free' for safety
const plan = (userRecord?.plan ?? 'free') as string;
const orgId = userRecord?.organizationId ?? userId;

// Surface gate
const boWeights = isSurfaceAvailable(plan, '<surface_id>')
  ? await get<SurfaceName>Weights(orgId)
  : undefined;

// After outcome is known:
if (boWeights?.source === 'bo') {
  void registerObservationSilently(
    '<surface_id>',
    orgId,
    boWeights.weights,     // or boWeights.params for non-weight surfaces
    targetValue,           // the outcome metric (0–1 or normalised 0–100)
    { ...metadata },
  );
}
```

The `void` prefix on `registerObservationSilently` is non-negotiable. Observation
registration is best-effort and must not block or throw into the HTTP response.

## Surface Designs

### Surface 1 — `content_scheduling`

**What it optimises:** The relative importance weights applied to each time-of-day
scoring dimension in `postingTimePredictor.getOptimalTimes()`. Currently the
predictor blends `historicalEngagement` from the user's past post data with
`industryBaseline` scores from `INDUSTRY_OPTIMAL_TIMES`. The BO surface controls
the blend weights plus three contextual multipliers that adjust the final score
for each candidate slot.

**Dimensions (5):**

| Parameter | Default | Min | Max | Rationale |
|-----------|---------|-----|-----|-----------|
| `historicalWeight` | 0.60 | 0.10 | 0.90 | Weight on user's own historical engagement data |
| `industryWeight` | 0.40 | 0.10 | 0.90 | Weight on industry baseline scores |
| `recencyBonus` | 0.10 | 0.00 | 0.30 | Bonus applied to slots used recently with high engagement |
| `peakHourMultiplier` | 1.20 | 1.00 | 1.50 | Score multiplier for peak-hour slots (9–11am, 12–2pm, 5–7pm) |
| `weekendDiscount` | 0.85 | 0.60 | 1.00 | Multiplier applied to Saturday/Sunday slots |

No `sumEquals` constraint. `historicalWeight` and `industryWeight` are treated
as independent weights (not required to sum to 1 — the service normalises them).

**Target metric:** `slot.score` (0–100) of the actual published post's time slot.
The API route at `app/api/optimize/auto-schedule/route.ts` already has the slot
scores in `platformResult.slots`. After the route schedules a post and gets the
final `engagementScore`, that value (0–100 normalised to 0–1) is the observation target.

**Integration point:** `lib/ml/posting-time-predictor.ts` — `getOptimalTimes()`.
The function already has a blend of historical and industry data. The BO weights
control the blend ratios. Pass weights as an optional parameter from the API route.

### Surface 2 — `backlink_scoring`

**What it optimises:** The weight distribution across backlink opportunity types
when ranking prospects in `analyzeOpportunities()`. Currently all prospect types
contribute equally — the final sort is purely by `domainAuthority` descending.
The BO surface controls per-type multipliers so high-converting opportunity types
float to the top.

**Dimensions (5):**

| Parameter | Default | Min | Max | Rationale |
|-----------|---------|-----|-----|-----------|
| `resourcePageWeight` | 0.25 | 0.05 | 0.50 | Weight for resource-page opportunities |
| `guestPostWeight` | 0.25 | 0.05 | 0.50 | Weight for guest-post opportunities |
| `brokenLinkWeight` | 0.20 | 0.05 | 0.50 | Weight for broken-link opportunities |
| `competitorLinkWeight` | 0.15 | 0.05 | 0.50 | Weight for competitor-link opportunities |
| `journalistMentionWeight` | 0.15 | 0.05 | 0.50 | Weight for journalist-mention opportunities |

Sum is not constrained — weights are used as per-type multipliers on the
`domainAuthority` score, not a probability distribution.

**Target metric:** The proportion of prospects that were actioned (contacted or
added to outreach). A proxy: `highValueCount / linksFound` (0–1). This is
already computed in `analyzeOpportunities()` and returned in `AnalysisResult`.

**Integration point:** `lib/backlinks/backlink-analyzer.ts` — `analyzeOpportunities()`.
The final sort currently ranks purely by `domainAuthority`. With BO weights,
replace the sort key with `domainAuthority * opportunityTypeWeight`. The
`app/api/backlinks/analyze/route.ts` route calls this function and has access
to the `orgId` for the BO lookup.

### Surface 3 — `authority_validation`

**What it optimises:** The claim-type priority order in `detectClaimType()` within
`lib/authority/claim-extractor.ts`. Currently the priority array is hardcoded as:
`['regulatory', 'statistical', 'temporal', 'causal', 'comparative', 'factual']`.
The BO surface provides a weight per claim type; `detectClaimType()` picks the
type with the highest pattern-match count weighted by the BO priority score.

**Dimensions (6):**

| Parameter | Default | Min | Max | Rationale |
|-----------|---------|-----|-----|-----------|
| `regulatoryPriority` | 0.30 | 0.05 | 0.50 | Priority for regulatory claim detection |
| `statisticalPriority` | 0.25 | 0.05 | 0.50 | Priority for statistical claim detection |
| `temporalPriority` | 0.15 | 0.05 | 0.50 | Priority for temporal claim detection |
| `causalPriority` | 0.15 | 0.05 | 0.50 | Priority for causal claim detection |
| `comparativePriority` | 0.10 | 0.05 | 0.50 | Priority for comparative claim detection |
| `factualPriority` | 0.05 | 0.05 | 0.50 | Priority for factual claim detection |

No `sumEquals` constraint — each is a priority multiplier, not a probability.

**Target metric:** `overallScore / 100` from `AuthorityAnalysisResult` (0–1).
This is returned by `analyzeAuthority()` and available in the API route.

**Integration point:** `lib/authority/claim-extractor.ts` — `detectClaimType()`.
The function currently uses a fixed `priority` array. With BO weights, the
function accepts an optional `priorityWeights` parameter and uses the weights
to sort candidate types by match-count multiplied by priority weight.
The `app/api/authority/analyze/route.ts` passes the weights down.

### Surface 4 — `psychology_levers`

**What it optimises:** The per-principle weights applied in `calculateOverallScore()`
within `PsychologyAnalyzer`. Currently `calculateOverallScore()` computes a
straight weighted average of `principle.strength * (principle.confidence / 100)`.
The BO surface adds a per-principle effectiveness multiplier that modulates the
contribution of each persuasion lever to the overall score.

**Dimensions (8):**

| Parameter | Default | Min | Max | Rationale |
|-----------|---------|-----|-----|-----------|
| `socialProofWeight` | 0.92 | 0.50 | 1.50 | Effectiveness multiplier for social-proof |
| `scarcityWeight` | 0.90 | 0.50 | 1.50 | Effectiveness multiplier for scarcity |
| `authorityWeight` | 0.88 | 0.50 | 1.50 | Effectiveness multiplier for authority |
| `reciprocityWeight` | 0.85 | 0.50 | 1.50 | Effectiveness multiplier for reciprocity |
| `lossAversionWeight` | 0.89 | 0.50 | 1.50 | Effectiveness multiplier for loss-aversion |
| `commitmentWeight` | 0.82 | 0.50 | 1.50 | Effectiveness multiplier for commitment |
| `likingWeight` | 0.80 | 0.50 | 1.50 | Effectiveness multiplier for liking |
| `anchoringWeight` | 0.86 | 0.50 | 1.50 | Effectiveness multiplier for anchoring |

Defaults mirror `effectivenessScore` values from `PSYCHOLOGY_PRINCIPLES` (divided
by 100 for normalisation, then scaled to the 0.50–1.50 range). No sum constraint.

**Target metric:** `analysis.overallScore / 100` (0–1). Available in the psychology
analyze route after `psychologyAnalyzer.analyzeContent()` completes.

**Integration point:** `lib/ai/psychology-analyzer.ts` — `calculateOverallScore()`.
The method accepts `principles` and `aiAnalysis`. The BO weights are applied as
per-principle multipliers when summing `strength * (confidence / 100)`. The
method needs an optional `leverWeights` parameter, defaulting to uniform 1.0.
The `app/api/psychology/analyze/route.ts` passes down the BO weights.

### Surface 5 — `self_healing_priority`

**What it optimises:** The ordering of healing issues returned by `detectIssues()`
in `lib/experiments/self-healer.ts`. Currently the issues are not explicitly
sorted by the service — they are returned in detection order. The BO surface
provides a weight per issue type; `detectIssues()` sorts the issues list
descending by the BO priority weight for the issue type before returning.

**Dimensions (8):**

| Parameter | Default | Min | Max | Rationale |
|-----------|---------|-----|-----|-----------|
| `missingMetaPriority` | 0.20 | 0.05 | 0.40 | Priority for missing title/meta issues |
| `brokenSchemaPriority` | 0.15 | 0.05 | 0.40 | Priority for broken schema issues |
| `lowGeoScorePriority` | 0.20 | 0.05 | 0.40 | Priority for low GEO score issues |
| `lowQualityScorePriority` | 0.15 | 0.05 | 0.40 | Priority for low quality score issues |
| `missingEntityPriority` | 0.10 | 0.05 | 0.40 | Priority for missing entity references |
| `shortTitlePriority` | 0.10 | 0.05 | 0.40 | Priority for short title tag issues |
| `missingH1Priority` | 0.05 | 0.05 | 0.40 | Priority for missing H1 issues |
| `weakMetaDescPriority` | 0.05 | 0.05 | 0.40 | Priority for weak meta description issues |

No `sumEquals` constraint. Defaults reflect the `critical`/`warning` severity
weighting already implicit in the existing rule order.

**Target metric:** An engagement proxy after a fix is applied: `1 - (criticalCount / issueCount)`.
Computed from the `issues` array returned by `analyzeForHealing()`. Higher = fewer
critical issues found, which indicates the priority order surfaced actionable items.

**Integration point:** `lib/experiments/self-healer.ts` — `detectIssues()` function.
The function currently does not sort the returned `issues` array. Add a sort at the
end of `detectIssues()` that uses BO priority weights. The function receives these
as an optional `priorityWeights` parameter. The API route
`app/api/experiments/healing/analyze/route.ts` passes the weights down.

### Surface 6 — `campaign_roi`

**What it optimises:** The platform budget-allocation weights in `calculateROI()`
within `ROIService`. Currently ROI is calculated per-platform independently
with no cross-platform weighting. The BO surface adds allocation weights that
represent the expected ROI multiplier for each platform — so platforms with
historically higher BO-optimised ROI float to the top of `byPlatform` results.

**Dimensions (8):**

| Parameter | Default | Min | Max | Rationale |
|-----------|---------|-----|-----|-----------|
| `youtubeAllocation` | 0.20 | 0.05 | 0.40 | Budget allocation weight for YouTube |
| `instagramAllocation` | 0.18 | 0.05 | 0.40 | Budget allocation weight for Instagram |
| `tiktokAllocation` | 0.15 | 0.05 | 0.40 | Budget allocation weight for TikTok |
| `twitterAllocation` | 0.15 | 0.05 | 0.40 | Budget allocation weight for Twitter/X |
| `facebookAllocation` | 0.12 | 0.05 | 0.40 | Budget allocation weight for Facebook |
| `linkedinAllocation` | 0.10 | 0.05 | 0.40 | Budget allocation weight for LinkedIn |
| `pinterestAllocation` | 0.05 | 0.05 | 0.40 | Budget allocation weight for Pinterest |
| `redditAllocation` | 0.05 | 0.05 | 0.40 | Budget allocation weight for Reddit |

No `sumEquals` constraint. Defaults are loosely aligned with average social
media ad spend distribution data.

**Target metric:** `overall.overallROI / 100` clamped to [0, 1]. If `overallROI`
is negative, use 0. Available in the ROI route after `calculateROI()` returns.

**Integration point:** `lib/roi/roi-service.ts` — `calculateROI()`. The method
currently sorts `byPlatform` by `roi` descending. With BO weights, multiply
each platform's `roi` value by its allocation weight before sorting — this
surfaces platforms the BO has learned are the highest-value allocation targets.
The `app/api/roi/route.ts` route calls `calculateROI()` and has access to
the `userId` to resolve `orgId` and `plan`.

**File**: `lib/bayesian/surfaces/content-scheduling.ts`

```typescript
/**
 * Bayesian Optimisation Surface Adapter — Content Scheduling
 *
 * Provides BO-optimised blend weights for the ML posting-time predictor.
 * Controls the ratio of historical vs. industry data and contextual multipliers.
 *
 * Surface: 'content_scheduling'
 * Dimensions: historicalWeight, industryWeight, recencyBonus, peakHourMultiplier, weekendDiscount
 *
 * @module lib/bayesian/surfaces/content-scheduling
 */

import { getOptimisedWeightsOrFallback } from '@/lib/bayesian/fallback';
import type { WeightResult } from '@/lib/bayesian/fallback';
import type { ParameterBounds } from '@/lib/bayesian/types';

export interface ContentSchedulingWeights {
  historicalWeight:     number;  // 0.10 – 0.90
  industryWeight:       number;  // 0.10 – 0.90
  recencyBonus:         number;  // 0.00 – 0.30
  peakHourMultiplier:   number;  // 1.00 – 1.50
  weekendDiscount:      number;  // 0.60 – 1.00
}

export interface ContentSchedulingResult {
  weights: ContentSchedulingWeights;
  source:  WeightResult['source'];
}

export const CONTENT_SCHEDULING_DEFAULTS: ContentSchedulingWeights = {
  historicalWeight:     0.60,
  industryWeight:       0.40,
  recencyBonus:         0.10,
  peakHourMultiplier:   1.20,
  weekendDiscount:      0.85,
};

export const CONTENT_SCHEDULING_BOUNDS: Record<keyof ContentSchedulingWeights, ParameterBounds> = {
  historicalWeight:     { min: 0.10, max: 0.90 },
  industryWeight:       { min: 0.10, max: 0.90 },
  recencyBonus:         { min: 0.00, max: 0.30 },
  peakHourMultiplier:   { min: 1.00, max: 1.50 },
  weekendDiscount:      { min: 0.60, max: 1.00 },
};

export async function getContentSchedulingWeights(
  orgId: string,
): Promise<ContentSchedulingResult> {
  const result = await getOptimisedWeightsOrFallback(
    'content_scheduling',
    orgId,
    () => ({ ...CONTENT_SCHEDULING_DEFAULTS }),
  );

  const raw = result.weights as Record<string, number>;
  const weights: ContentSchedulingWeights = {
    historicalWeight:     raw.historicalWeight     ?? CONTENT_SCHEDULING_DEFAULTS.historicalWeight,
    industryWeight:       raw.industryWeight       ?? CONTENT_SCHEDULING_DEFAULTS.industryWeight,
    recencyBonus:         raw.recencyBonus         ?? CONTENT_SCHEDULING_DEFAULTS.recencyBonus,
    peakHourMultiplier:   raw.peakHourMultiplier   ?? CONTENT_SCHEDULING_DEFAULTS.peakHourMultiplier,
    weekendDiscount:      raw.weekendDiscount      ?? CONTENT_SCHEDULING_DEFAULTS.weekendDiscount,
  };

  return { weights, source: result.source };
}
```

## Risk Notes

**Risk 1 — `posting-time-predictor.ts` internal blend logic**
The ML predictor may not have explicit `historicalWeight` / `industryWeight`
variables as scalar floats. Read the full file before Task 7. If the blend
is implemented differently (e.g. a data-points count threshold), the
integration design for `content_scheduling` will need to adapt. The surface
adapter (Task 1) is unaffected — only the integration (Task 7) may need adjustment.

**Risk 2 — `psychology-analyzer.ts` uses a class (not standalone functions)**
The `calculateOverallScore()` method is private to `PsychologyAnalyzer`. The
`leverWeights` parameter must be threaded through the public `analyzeContent()`
call. Do not change the class constructor or make `calculateOverallScore()` public.
Optional parameters on existing public methods are backward-compatible.

**Risk 3 — `authority-analyzer.ts` call chain for `priorityWeights`**
`analyzeAuthority()` calls `extractClaims()` which calls `detectClaimType()`.
Three files need modification for Task 9. Read all three files before implementing
to confirm the exact call chain, then thread the weights as an optional parameter
through each layer.

**Risk 4 — `campaign_roi` platform name normalisation**
The `platform` field in `ContentInvestment` is a free-text string from the DB.
It may be stored in various formats (`YouTube`, `youtube`, `YOUTUBE`). Apply
`.toLowerCase()` before the `PLATFORM_TO_WEIGHT` lookup to avoid cache misses.

**Risk 5 — Scale plan string casing**
`BO_PLAN_LIMITS` keys are lowercase (`scale`, `growth`, `pro`, `free`). If the
DB stores `Scale` or `SCALE`, the `isSurfaceAvailable()` lookup will fall back
to `free` (see `getBOFeatureLimits()` line 76). Apply `.toLowerCase()` to the
plan string before calling `isSurfaceAvailable()`.

**Risk 6 — `auto-schedule/route.ts` Prisma import**
The auto-schedule route uses Supabase directly for data operations and may not
import Prisma. The user plan lookup needs Prisma. Read the file before Task 7
and add the import only if it is not already present.
