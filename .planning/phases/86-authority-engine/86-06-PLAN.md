  meta: { title?: string; description?: string; viewport?: string };

**heading-hierarchy.ts**: Analyses H1→H2→H3 progression.
- `analyseHeadingHierarchy(content: PageContent): { score: number; issues: DesignIssue[] }`
- Extract headings from PageContent.headings array
- Check: exactly 1 H1 (penalty if 0 or >1)
- Check: no level skips (H1→H3 without H2 = -10 pts)
- Check: logical progression (H2s under H1, H3s under H2)
- Check: consistent spacing between heading levels
- Max score: 15 points

**content-structure.ts**: CITABLE framework compliance analysis.
- `analyseContentStructure(content: PageContent): { claimIsolation: number; tableListStructure: number; answerQueryAlignment: number; boldEntityDefinitions: number; issues: DesignIssue[] }`
- **Claim Isolation (0-15)**: Paragraph length analysis — ideal is 25-40 words per fact. Penalise >100 word paragraphs without breaks.
- **Table/List Structure (0-15)**: Count tables + ordered/unordered lists. Score: 0 = 0pts, 1-2 = 8pts, 3+ = 15pts.
- **Answer-Query Alignment (0-15)**: Detect FAQ patterns (Q&A blocks, question headings, "What is...", "How to...", "Why..." patterns). Score based on count.
- **Bold Entity Definitions (0-10)**: Detect patterns like `<strong>Term</strong>: definition` or `**Term** — description`. Score based on presence.

AVOID: Don't use external APIs for content analysis — this is all local NLP/pattern matching. Don't import from files that don't exist yet.

**cro-signals.ts**: CRO readiness analysis.
- `analyseCROSignals(content: PageContent): { score: CROReadinessScore; issues: DesignIssue[] }`
- **Conversion Funnel (0-20)**: Count CTAs (buttons with action text: "buy", "sign up", "get started", "contact", "request", "book"). Score: 0 = 0pts, 1-2 = 12pts, 3-5 = 20pts, >5 = 15pts (too many = friction).
- **Trust Signals (0-20)**: Detect author bio patterns, review/testimonial sections, credential mentions, social proof indicators, security badges. Score per signal found.
- **Friction Reduction (0-20)**: Assess form complexity (fields count), choice overload (menu items), CTA clarity (distinct vs ambiguous). Fewer fields = higher score.
- **Mobile Conversion (0-20)**: Check if CTAs are in mobile-friendly positions, clickable phone numbers, sticky elements detected.
- **Above-Fold Conversion (0-20)**: First 3 headings + first CTA position. Score high if headline, value prop, and CTA appear early.

**mobile-readiness.ts**: Mobile-specific checks.
- `analyseMobileReadiness(content: PageContent): { score: number; issues: DesignIssue[] }`
- Check viewport meta tag exists (`content: PageContent.meta.viewport`)
- Font size analysis: body text should be >=16px (detect `font-size` in inline styles, or flag if not specified)
- Touch target sizing: buttons/links should be >=44px (heuristic based on text length and surrounding spacing)
- Horizontal scroll risk: detect tables without responsive wrapper, images without max-width
- Max score: 15 points

**performance-checker.ts**: PageSpeed Insights API wrapper.
- `checkPerformance(url: string): Promise<{ score: number; metrics: CoreWebVitals; issues: DesignIssue[] }>`
- API: `https://www.googleapis.com/pagespeedonline/v5/runPagespeed?url=${url}&strategy=mobile`
- No API key required for basic use
- Extract: LCP (target <=2.5s), INP (target <=200ms), CLS (target <=0.1), performance score
- Normalise to 0-15 scale: performance score 90-100 = 15, 50-89 = 10, 25-49 = 5, 0-24 = 0
- Handle API failures gracefully: return score 0 with warning issue, don't throw
- Add timeout (10s) to prevent hanging on slow API responses

```typescript
export interface CoreWebVitals {
  lcp: number; // seconds
  inp: number; // milliseconds
  cls: number; // score
  performanceScore: number; // 0-100
  accessibilityScore: number; // 0-100
}
```

AVOID: Don't install new packages. Don't make PageSpeed API mandatory — return defaults if URL not provided or API fails. Use native fetch() for API calls.

**llm-citation-fitness.ts**: Full CITABLE framework composite scorer.
- `analyseLLMCitationFitness(content: PageContent): { score: LLMCitationFitnessScore; issues: DesignIssue[] }`
- Combines signals from content-structure (C, T, A, B) with:
  - **Inline Attribution (0-15)** (I): Detect citation patterns — "according to", "source:", "(Author, Year)", "[1]", footnote markers. Score by count.
  - **Logical Heading Depth (0-15)** (L): Reuse heading-hierarchy score, normalised to 0-15
  - **Entity Consistency (0-15)** (E): Import `analyseEntityCoherence` from `lib/geo/entity-analyzer.ts` (Phase 85). Run entity analysis on content text, normalise entity coherence score to 0-15.
- Composite total: sum all 7 CITABLE dimensions

**design-analyzer.ts**: Main orchestrator that runs all design checks.
- `analyseDesign(input: { content?: string; url?: string; html?: string }): Promise<DesignAuditResult>`
- **Input parsing**:
  1. If `url` provided: fetch page HTML via `fetch(url)` + parse into PageContent structure (extract headings, paragraphs, links, images, tables, lists, forms, meta from raw HTML using regex/string parsing — no jsdom needed for basic extraction)
  2. If `html` provided: parse directly into PageContent
  3. If `content` (plain text) provided: convert to basic PageContent (headings from markdown # syntax, paragraphs from double newlines)
- **Run all analysers in parallel**:
  1. `analyseHeadingHierarchy(pageContent)`
  2. `analyseContentStructure(pageContent)`
  3. `analyseCROSignals(pageContent)`
  4. `analyseMobileReadiness(pageContent)`
  5. `checkPerformance(url)` — only if URL provided, otherwise skip with score 0
  6. `analyseLLMCitationFitness(pageContent)`
- **Compose results**: Merge all scores into DesignAuditResult
- **Overall score**: Weighted average — Design Quality 35%, CRO Readiness 30%, LLM Citation Fitness 35%
- **Generate recommendations**: Based on lowest-scoring areas, generate top 5 actionable recommendations

HTML parsing helper: create a simple `parseHTML(html: string): PageContent` function that extracts elements using regex patterns (not a full DOM parser). This is sufficient for heading, paragraph, link, image, table, list, and form detection.

AVOID: Don't install jsdom or cheerio — use regex-based HTML parsing for this initial version. Don't make URL fetching mandatory. Don't throw if any individual analyser fails — return partial results.
