# Plan 77-04: Multi-Platform Publish & Post Status Tracking

## objective

Enable publishing a single piece of content to multiple platforms simultaneously and provide real-time status tracking across the entire content lifecycle. Currently, a post targets exactly one platform. Users who want to cross-post must generate/schedule separately for each platform. The cross-posting infrastructure exists (`/api/content/cross-post`, `lib/ai/content-repurposer.ts`) but is not wired into the primary content creation flow. This plan adds multi-platform selection to the composer, adapts content per platform using the repurposer, creates individual scheduled posts per platform, and adds a post status tracker showing the state of each platform publish.

Linear: SYN-45
Phase dir: `.planning/phases/77-content-creation-flow/`

## context

### Discovery Findings

**Gap 1 -- Single platform per post**: The content page has a single platform dropdown. Publishing to 3 platforms requires 3 separate generate/schedule cycles. This contradicts the platform's "post once, publish everywhere" value proposition (Phase 40).

**Gap 2 -- No cross-post in composer**: The cross-post page exists at `/dashboard/content/cross-post` but is a separate flow, not integrated into the main content creation path. Users must navigate to a different page after generating content.

**Gap 3 -- No post status tracking**: After scheduling, users have no visibility into whether their post was published successfully, failed, or is still pending on each platform. The schedule page shows a single status, but multi-platform posts need per-platform status.

### Architecture Decision

- **Multi-platform selector**: Replace the single `<select>` with a multi-select checkbox group in `GenerationSettings`
- **Primary + secondary platforms**: Generate content for the "primary" platform, then use the content repurposer to adapt for secondary platforms
- **One scheduled Post per platform**: Each platform gets its own `Post` record in the DB (this aligns with the existing scheduler and publish cron architecture which processes posts individually)
- **Batch ID**: Link related cross-platform posts via a shared `batchId` in `Post.metadata` so the UI can group them
- **Post status component**: `PostStatusTracker` component shows a row per platform with its status (pending, scheduled, publishing, published, failed)

### Task 2: Content adaptation for secondary platforms

**File**: `app/dashboard/content/page.tsx`

When `selectedPlatforms.length > 1`, after generating content for the primary platform:

1. Call the content repurposer to adapt for each secondary platform:
   ```ts
   // After primary content is generated:
   if (selectedPlatforms.length > 1) {
     const adaptations = await fetch('/api/content/cross-post', {
       method: 'POST',
       credentials: 'include',
       headers: { 'Content-Type': 'application/json' },
       body: JSON.stringify({
         content: generatedContent.primary,
         sourcePlatform: platform,
         targetPlatforms: selectedPlatforms.filter(p => p !== platform),
       }),
     });
   }
   ```

2. Store adaptations in state: `const [platformAdaptations, setPlatformAdaptations] = useState<Record<string, string>>({});`

3. Show adapted content in the preview area (from 77-02) -- one preview per selected platform, scrollable horizontally or stacked vertically

First read `/api/content/cross-post/route.ts` to understand its request/response shape.

**Verification**: Generate content with 3 platforms selected -- see adapted content for each platform.

   ```ts

         content: platformAdaptations[targetPlatform] || primaryContent,
         platform: targetPlatform,
   ```

### Task 5: Add batchId filter to scheduler API

**File**: `app/api/scheduler/posts/route.ts`

Minor update to the GET handler:

1. Add `batchId` to the query schema: `batchId: z.string().optional()`
2. When `batchId` is provided, add a metadata JSON filter:
   ```ts
   if (batchId) {
     where.metadata = { path: ['batchId'], equals: batchId };
   }
   ```

**Note**: Prisma JSON filtering syntax depends on the database. For PostgreSQL: `where: { metadata: { path: ['batchId'], equals: batchId } }`. Check the Prisma docs for the exact syntax.

**Verification**: `GET /api/scheduler/posts?batchId=batch-xxx` returns only posts in that batch.
