# Plan 80-01: Brand Profile API

## objective

Add a dedicated `GET /api/brand-profile` and `PATCH /api/brand-profile` endpoint that reads and writes the current user's organisation brand identity. Currently no user-facing route exists for updating organisation fields post-onboarding — the only edit path is `PATCH /api/organizations/[orgId]` which requires org admin status and is designed for admin operations. This plan creates a simple self-service brand profile endpoint scoped to the caller's active organisation.

Linear: SYN-55
Phase dir: `.planning/phases/80-brand-profile-setup/`

- Organisation model: `prisma/schema.prisma` — `Organization` has `name`, `description`, `website`, `industry`, `teamSize`, `abn`, `logo`, `primaryColor`, `favicon`, `socialHandles`, `aiGeneratedData`
- Auth pattern: `lib/auth/jwt-utils.ts` — `getUserIdFromRequestOrCookies(request)` + `getUserIdFromRequest(request)`
- Active org lookup: `User.organizationId` or `User.activeOrganizationId` — join via Prisma

- Response pattern: `NextResponse.json({ data: ... })` (success) / `NextResponse.json({ error: '...' }, { status: N })` (failure)
- Org update pattern: `app/api/organizations/[orgId]/route.ts` PATCH — reference for field allowlist and audit log

## context

### Discovery Findings

**Gap 1 — No self-service brand edit API**: After onboarding, users cannot update their org's `logo`, `primaryColor`, `favicon`, `description`, `website`, `industry`, `teamSize`, `abn`, or `socialHandles` from any user-facing UI. The onboarding route (`POST /api/onboarding`) saves these on initial setup but has no re-entry point. The admin route (`PATCH /api/organizations/[orgId]`) requires checking `settings.admins` array which new users won't be in.

**Gap 2 — No brand profile read endpoint**: There is no route that returns a user's own org brand fields in a clean shape for display in a profile UI card.

**Gap 3 — Logo is URL-only**: The `Organization.logo` field stores a URL string. To support logo uploads, the API should accept both a URL string and a Supabase Storage upload path. The existing `POST /api/media/upload` handles the actual file upload and returns a URL — the brand profile PATCH just stores the returned URL.

### Architecture Decision

- New file: `app/api/brand-profile/route.ts`
- Auth: `getUserIdFromRequestOrCookies(request)` — same as all user-facing routes
- Active org resolution: lookup `User.organizationId` (current active org) → if null, return 404 with helpful message
- GET: return brand-profile shape from org fields
- PATCH: validate with Zod, update only supplied fields, no audit log (non-destructive user self-service), return updated shape
- No `PLAN_LIMITS` or admin checks — any org member can edit their own org's brand identity
- Favicon and logo: URL-only. The media upload endpoint handles the file → URL conversion separately

## tasks

### Task 1: Create `app/api/brand-profile/route.ts`

**File**: `app/api/brand-profile/route.ts`

**Zod schema:**

```ts
const brandProfileSchema = z.object({
  name: z.string().min(1).max(100).optional(),
  description: z.string().max(500).optional().or(z.literal('')),
  website: z.string().url().max(200).optional().or(z.literal('')),
  industry: z.string().max(100).optional(),
  teamSize: z.string().max(50).optional(),
  abn: z.string().max(20).optional().or(z.literal('')),
  logo: z.string().url().max(500).optional().or(z.literal('')),
  favicon: z.string().url().max(500).optional().or(z.literal('')),
  primaryColor: z.string().regex(/^#[0-9A-Fa-f]{6}$/).optional().or(z.literal('')),
  socialHandles: z.record(z.string().max(100)).optional(),
});
```

**BrandProfileResponse shape** (returned by both GET and PATCH):

```ts
interface BrandProfileResponse {
  id: string;
  name: string;
  slug: string;
  description: string | null;
  website: string | null;
  industry: string | null;
  teamSize: string | null;
  abn: string | null;
  logo: string | null;
  favicon: string | null;
  primaryColor: string | null;
  socialHandles: Record<string, string>;
  updatedAt: string; // ISO string
}
```

**GET handler:**

1. Auth: `getUserIdFromRequestOrCookies(request)` — return 401 if missing
2. Lookup user with `organizationId`:
   ```ts
   const user = await prisma.user.findUnique({
     where: { id: userId },
     select: { organizationId: true },
   });
   ```
3. If `user.organizationId` is null, return `{ error: 'No organisation found. Complete onboarding first.' }` with 404
4. Fetch org:
   ```ts
   const org = await prisma.organization.findUnique({
     where: { id: user.organizationId },
     select: { id, name, slug, description, website, industry, teamSize, abn, logo, favicon, primaryColor, socialHandles, updatedAt },
   });
   ```
5. Return `{ data: brandProfileResponse }` with 200

**PATCH handler:**

1. Auth + org lookup (same as GET)
2. Parse body: `request.json()` with Zod validation — return 400 on failure
3. Build `updateData` — only include fields that are present in the validated body:
   ```ts
   const updateData: Record<string, unknown> = {};
   if (body.name !== undefined) updateData.name = body.name;
   if (body.description !== undefined) updateData.description = body.description || null;
   if (body.website !== undefined) updateData.website = body.website || null;
   if (body.industry !== undefined) updateData.industry = body.industry;
   if (body.teamSize !== undefined) updateData.teamSize = body.teamSize;
   if (body.abn !== undefined) updateData.abn = body.abn || null;
   if (body.logo !== undefined) updateData.logo = body.logo || null;
   if (body.favicon !== undefined) updateData.favicon = body.favicon || null;
   if (body.primaryColor !== undefined) updateData.primaryColor = body.primaryColor || null;
   if (body.socialHandles !== undefined) updateData.socialHandles = body.socialHandles;
   ```
4. Update:
   ```ts
   const updated = await prisma.organization.update({
     where: { id: user.organizationId },
     data: updateData,
     select: { id, name, slug, description, website, industry, teamSize, abn, logo, favicon, primaryColor, socialHandles, updatedAt },
   });
   ```
5. Return `{ data: brandProfileResponse }` with 200
