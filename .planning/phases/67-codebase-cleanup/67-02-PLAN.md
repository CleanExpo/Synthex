<task type="auto">
  <name>Task 1: Audit API routes for frontend callers</name>
  <files>— (read-only audit, no file changes)</files>
  <action>
    For each directory under app/api/, determine if a frontend caller exists.

    Method — for each route file, extract the URL path and search frontend code:

    Step 1: List all route.ts files
      find app/api -name "route.ts" | sort

    Step 2: For each route URL (e.g. /api/content-drafts), search:
      grep -rn "/api/content-drafts\|content-drafts" hooks/ components/ app/dashboard/ --include="*.ts" --include="*.tsx"

    Classify each route as:
      WIRED    — at least one frontend caller found (hook, component, or page)
      INTERNAL — intentionally server-only (cron, webhook, admin, auth, monitoring, health)
      ORPHANED — no frontend caller, not intentionally server-only

    Produce a mental list of ORPHANED routes to act on in Task 2.

    Priority routes to check (from Sprint 3 scan — most likely orphans):
    - /api/content-drafts
    - /api/features
    - /api/businesses
    - /api/clients
    - /api/eeat
    - /api/example
    - /api/cache
    - /api/backup
    - Any others with no obvious hook in hooks/

    Skip: /api/cron/*, /api/webhooks/*, /api/auth/*, /api/admin/*, /api/monitoring/*, /api/health
  </action>
  <verify>
    Audit complete — ORPHANED list identified and ready for Task 2.
  </verify>
  <done>Every app/api/ route classified as WIRED, INTERNAL, or ORPHANED.</done>
</task>

<task type="auto">
  <name>Task 2: Wire or document each orphaned route</name>
  <files>Varies based on Task 1 findings</files>
  <action>
    For each ORPHANED route identified in Task 1, apply one of these actions:

    **Action A — Wire it (if the route serves useful UI data):**
    - Check if a hook already exists in hooks/ that calls this endpoint
    - If hook exists but no component uses it: add the component or wire to existing page
    - If no hook exists: create a minimal hook in hooks/ following use-api.ts pattern
    - Do NOT create new dashboard pages — wire to existing pages or widgets only

    **Action B — Mark as intentional internal route (if it's legitimately server-only):**
    Add a JSDoc comment at the top of the route.ts file:
    ```typescript
    /**
     * @internal Server-only endpoint — not called directly by frontend UI.
     * Used by: [explain: admin tooling / background jobs / external integrations / etc]
     */
    ```

    **Action C — Note as candidate for removal (if it's truly dead code):**
    Add JSDoc:
    ```typescript
    /**
     * @deprecated No active callers. Candidate for removal in a future cleanup phase.
     * Last reviewed: 2026-03-03
     */
    ```
    Do NOT delete routes in this plan — only mark them. Deletion requires explicit user review.

    Apply the most appropriate action to each orphaned route found in Task 1.
    Prioritise Action A for routes that clearly map to existing UI features.
    Use Action B/C for routes with no obvious UI home.

    After all changes: run npm run type-check
  </action>
  <verify>
    npm run type-check (0 errors)
    npm test (≥1506 passing, no regressions)
  </verify>
  <done>
    All previously-orphaned routes are either:
    (a) wired to a frontend caller, OR
    (b) documented with @internal or @deprecated JSDoc
    Type-check and tests pass.
  </done>
</task>
