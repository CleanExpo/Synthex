**Current data fetching landscape:**

| Pattern | Package | Where used | When to use |
|---------|---------|------------|-------------|
| Custom useApi() | hooks/use-api.ts (internal) | hooks/use-dashboard.ts, 30+ existing hooks | Existing hook-based data fetching in hooks/ |
| Real SWR | swr ^2.2.0 | GamificationWidget, ContentSuggestionsWidget, InsightsWidget, WorkflowsPageClient, IntelligencePanel, ReviewQueuePanel, use-workflow-executions | Standalone widget components (v2.0+ pattern) |
| Raw fetch() | browser Fetch API | dashboard page (app/dashboard/page.tsx fetchDashboardData), some page-level components | ONLY for authenticated SSR/RSC requests, never in client components |

**Canonical standard to enforce (decision for CLAUDE.md):**
- Hooks in hooks/: use hooks/use-api.ts (useApi, useMutation) — do NOT migrate existing
- Standalone widgets/components: use real SWR (useSWR with credentials: 'include' fetcher)
- Raw fetch() in 'use client' components: migrate to SWR
- Raw fetch() in API routes or server actions: fine as-is (server-side)

**Architecture rules:**
- All client fetch calls MUST use credentials: 'include' (httpOnly auth-token cookie)
- Never introduce new custom fetching abstractions
- Auth: Supabase ONLY

  <name>Task 2: Migrate top candidates to SWR pattern</name>
  <files>Up to 5 component files identified in Task 1</files>
  <action>
    For each migration candidate, replace raw fetch() with useSWR following the
    established pattern from GamificationWidget.tsx and InsightsWidget.tsx:

    **SWR standard pattern (copy from GamificationWidget):**
    ```typescript
    import useSWR from 'swr';

    const fetchJson = async (url: string) => {
      const res = await fetch(url, { credentials: 'include' });
      if (!res.ok) return null;
      return res.json();
    };

    // In component:
    const { data, error, isLoading } = useSWR<ResponseType>(
      '/api/endpoint',
      fetchJson,
      { revalidateOnFocus: false }
    );

    // Render nothing if no data (graceful degradation):
    if (!data || error) return null;
    ```

    Rules:
    - ALWAYS use `credentials: 'include'` in the fetcher (httpOnly cookie auth)
    - Use `revalidateOnFocus: false` for data that doesn't need real-time updates
    - Return null (not error state) for optional dashboard widgets
    - Keep error state for required page-level data

    After each migration, run: npm run type-check
    Fix any TypeScript errors before moving to the next file.

    If a candidate is too complex to migrate safely (multiple interdependent fetches,
    complex error handling, optimistic updates), skip it and note it in the summary.
    Safety > thoroughness for a v3.0 launch prep phase.
