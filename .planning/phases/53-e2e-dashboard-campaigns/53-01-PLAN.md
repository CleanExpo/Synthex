## Key Findings (Pre-Run Analysis)

### Auth architecture (middleware.ts)
The middleware at `C:\Synthex\middleware.ts` protects `/dashboard/*` paths by checking:
1. Supabase session (httpOnly cookie from Supabase SSR)
2. OR `auth-token` cookie (any non-empty value)

This means setting `auth-token = "test-token"` in a cookie bypasses the middleware
redirect. This is the auth strategy for E2E tests.

### Why current dashboard tests fail
- `dashboard.spec.ts` and `dashboard-tabs.spec.ts` use `mockAuthentication()` which
  sets `localStorage.synthex_token` — ignored by both middleware and Supabase auth
- `createAuthenticatedContext()` in `auth.fixture.ts` tries:
  1. POST `/api/auth/dev-login` → 404 (endpoint doesn't exist)
  2. POST `/api/auth/login` → likely 404 (app uses `/api/auth/unified-login`)
  Returns `false` without setting any cookie → no auth

### Missing /dashboard/campaigns page
- `CampaignPage.goto()` navigates to `/dashboard/campaigns`
- This page does not exist in the codebase (confirmed via Glob)
- Without auth cookie: middleware redirects to `/login`
- Tests that check for campaign content → fail

### Static HTML routes in old tests
- `navigation.spec.ts`: `/pricing.html`, `/demo.html`, `/docs.html` → 404
- `dashboard.spec.ts`: `/dashboard.html` → 404
- `staging.spec.ts`: `/login.html`, `/dashboard.html` → 404

### Staging-environment tests
- `staging.spec.ts` checks for `x-environment: staging` response header
- Checks `window.FEATURE_FLAGS` (not in codebase)
- Checks `/api/v1/config` endpoint (doesn't exist)
- These can never pass in dev environment
